# 进化论 · 虚函数与抽象基类

前面我们接触到了 `virtual` 关键字——打了这个标记的成员函数，会在运行时根据对象的**真实类型**（而不是引用/指针的声明类型）决定到底执行哪个版本。这个机制叫做**动态绑定**。

这节我们把虚函数的工作原理打开来看清楚，以及它的进化版本——纯虚函数和抽象基类。

------

**虚函数到底做了什么**

```cpp
class Animal
{
public:
    virtual void speak() { cout << "某种叫声" << endl; }
};

class Cat : public Animal
{
public:
    void speak() override { cout << "喵" << endl; }
};

class Dog : public Animal
{
public:
    void speak() override { cout << "汪" << endl; }
};

void makeSound(Animal& a)
{
    a.speak();
}

Cat cat;
Dog dog;
makeSound(cat);   // 喵
makeSound(dog);   // 汪
```

`makeSound` 函数接收的是 `Animal&`，但调用 `speak` 时却执行了各自派生类的版本。它是在运行时根据对象的真实类型，从**虚函数表**中查到正确的函数地址然后去执行的。

每个含有虚函数的类的对象，内部都会隐藏一个指针（vptr），指向该类共享的一张虚函数表（vtable）。vtable 里面记录着各个虚函数的实际入口地址。当通过基类指针/引用调用虚函数时，程序按照 vptr → vtable 的路径找到对应的确确实实的函数来执行。这就是 C++ 动态多态的底层机制。

------

**override：你的安全卫士**

如果你打算覆盖基类的虚函数，写法上加一个 `override`：

```cpp
class Cat : public Animal
{
public:
    void speak() override     // ✅ 正确覆盖
    {
        cout << "喵" << endl;
    }
};
```

但如果基类没有这个虚函数，或者你写错了签名：

```cpp
// void Speek() override;    // ❌ 编译错误——基类没有虚函数 Speek，override 能帮你发现笔误
```

`override` 不是语法糖——它是编译器的检查点。没有它，你在基类函数名字上多打个字母，它就变成了全新的、不相关的函数，多态就静默地失效了，还不会报任何错误。

再就是 C++ 允许你"跨层"覆盖——比如 Base → Mid → Leaf 三层，Leaf 可以直接覆盖 Base 声明的虚函数。`override` 在这种情况下是半必须的：一眼就能看出这个函数是在重写最古老的根类里的虚函数。

所以，如果你在 override 一个虚函数，就写上 `override`。

------

**final：禁止再被覆盖**

如果你确定某个虚函数不应该再被任何子类覆盖：

```cpp
class Cat : public Animal
{
public:
    void speak() override final {}   // Cat 及以后的类不能再覆盖 speak
};
```

`final` 也可以在类上使用：`class Cat final {}` 会阻止任何类继续从 Cat 继承。

------

**纯虚函数与抽象基类**

有些基类提供"概念上的共享接口"，但自己不去实现它——因为具体行为只能由派生类定义。这种基类就是**抽象基类**，实现这种"纯接口"靠的是**纯虚函数**：

```cpp
class Shape          // 抽象基类——不能被直接实例化
{
public:
    virtual double area() const = 0;    // 纯虚函数——没有实现，只有声明
    virtual ~Shape() = default;
};
```

`= 0` 就是说"我只是个接口，实现由派生类自己看着办"。任何包含纯虚函数的类都成了**抽象类**，不能直接创建对象。尝试 `Shape s;` 会直接编译报错。

```cpp
class Circle : public Shape
{
    double radius;
public:
    Circle(double r) : radius(r) {}
    double area() const override { return 3.14159 * radius * radius; }
};

class Rectangle : public Shape
{
    double width, height;
public:
    Rectangle(double w, double h) : width(w), height(h) {}
    double area() const override { return width * height; }
};
```

现在我们可以这样用：

```cpp
vector<unique_ptr<Shape>> shapes;
shapes.push_back(make_unique<Circle>(5.0));
shapes.push_back(make_unique<Rectangle>(3.0, 4.0));

for (const auto& s : shapes)
    cout << "面积：" << s->area() << endl;
```

每个 Shape 的具体子类给出自己的 area 算法，上层代码通过基类接口调用，不用管下面到底是什么图形。（我们学过的多态在迭代器上再次现身——`s` 的类型是 `const unique_ptr<Shape>&`，但解出后的 `s->area()` 遵循对象真实的 Circle / Rectangle 类型决定调用哪个版本。）

------

**接口类：纯虚函数的集大成**

如果一个类所有成员函数都是纯虚的，没有成员变量，那它就是**接口类**。C++ 没有像 Java 那样的 `interface` 关键字，但用抽象类可以做出完全相同的事：

```cpp
class IDrawable
{
public:
    virtual void draw() const = 0;
    virtual ~IDrawable() = default;
};
```

任何实现这个接口的类，必须提供 `draw()` 的定义。通过 `IDrawable*` 或 `IDrawable&` 就能用多态来统一擦偶各种各样的可绘制对象。

------

**虚函数的开销**

虚函数的动态分派不是免费的——每次调用会多一次间接寻址（查 vtable），而且对象多了一个 vptr（通常是 8 个字节的额外空间）。但对于绝大多数程序，这点开销完全可以忽略。除非你在几十亿次循环调用里做微秒级的优化，虚函数的开销才值得拿出来计较。

别在没有证据的情况下"优化掉"虚函数——它带来的代码清晰度和可维护性效益，远比几纳秒的间接寻址重要。
