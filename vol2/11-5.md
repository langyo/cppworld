# 偷懒的诀窍 · 合成函数与虚析构

编译器的脾气你真得摸透。你不给类写构造函数、析构函数、拷贝函数、移动函数，编译器会悄悄帮你生成一套。这套叫**合成函数**。绝大多数时候它们挺好，但到了继承场景，只要一步没摸对，沉默的隐患就埋下了。

------

**编译器会默默帮你生成哪些函数**

如果你没有声明，编译器会自动生成以下六个：

1. 默认构造函数
2. 析构函数
3. 拷贝构造函数
4. 拷贝赋值运算符
5. 移动构造函数（C++11）
6. 移动赋值运算符（C++11）

生成规则是"逐个成员"做它们各自该做的事——对整型成员不做任何事（留垃圾值），对类类型成员调用它们对应的构造/析构。

但凡你手动声明了其中任何一个，编译器生成另外一些的行为会受影响（这有一张很长的表格，需要的时候查阅即可）。最实用的一条：**你只要声明了一个析构函数，编译器就不再为你生成默认的移动构造函数和移动赋值运算符。** 这可能导致你的容器在移动场景下悄悄地降级为"拷贝 + 析构"而没有任何警告。

解决办法是用 `= default` 把它们显式"召唤"回来：

```cpp
class MyClass
{
public:
    ~MyClass() { /* 自定义清理 */ }
    MyClass(MyClass&&) = default;              // 把移动构造要回来
    MyClass& operator=(MyClass&&) = default;   // 把移动赋值也要回来
};
```

------

**虚析构函数的真相**

前面我们提过一句：**任何作为基类的类，都应该有一个虚析构函数。** 这里把问题讲清楚。

假如你没有写虚析构：

```cpp
class Base
{
public:
    ~Base() { cout << "Base 析构" << endl; }
};

class Derived : public Base
{
private:
    int* data;
public:
    Derived() : data(new int[100]) {}
    ~Derived() { delete[] data; cout << "Derived 析构" << endl; }
};

// 如果这样用：
Base* ptr = new Derived();
delete ptr;      // 只输出 "Base 析构"——data 泄漏了，100 个 int 的数组永远丢失！
```

因为 `ptr` 的静态类型是 `Base*`，编译器按照这个类型去调析构函数。`Base` 的析构不是虚的，所以只调了 `Base` 的，Derived 的析构没被触发。

改法——往基类析构上加 `virtual`：

```cpp
class Base
{
public:
    virtual ~Base() = default;
};
```

添加 `virtual` 后，即使通过基类指针 `delete`，也能正确调用派生类的析构，派生类的资源得到干净释放。

------

**`= default` vs `= delete`**

- `= default`："编译器，帮我生成默认版本吧，我只要它行为很标准就好。"
- `= delete`："明确禁止——谁要是调这个，编译直接报错。"

常见用法：禁止拷贝。

```cpp
class NonCopyable
{
public:
    NonCopyable() = default;

    NonCopyable(const NonCopyable&) = delete;            // 禁止拷贝
    NonCopyable& operator=(const NonCopyable&) = delete;

    NonCopyable(NonCopyable&&) = default;                // 但允许移动
    NonCopyable& operator=(NonCopyable&&) = default;
};
```

很多管理独占资源的类（如 unique_ptr、文件句柄）都应该是可移动但不可拷贝的。先声明移动，再 `= delete` 拷贝，这是标准写法。

------

**三五法则**

C++ 社区有个耳熟能详的口诀——**三/五法则**：

如果你手动定义了以下任一函数：
- 析构函数
- 拷贝构造函数
- 拷贝赋值运算符
- 移动构造函数
- 移动赋值运算符

那么你很可能需要把全部五个（前三个必须得是至少三个）都理清楚——因为你正在管理某种"不普通的资源"，编译器默认的逐个成员行为在资源共享/所有权方面可能坑你。

建议的顺序：先确定资源是谁的、怎么转移；再根据资源的性质决定是可拷贝的、可移动的还是两者都有；然后把对应的五个函数依次写好（或用 `= default` 和 `= delete` 迅速表达意图）。虚析构这块尤其别偷懒——但凡基类可能被多态 delete，就把析构标 `virtual`，不给自己留坑。
