# 特殊人格 · 模板特例化

模板的哲学是"一套代码对付所有类型"。但有时候，某个特定类型需要不同的处理逻辑——比如 `vector<bool>` 为了省内存用位压缩存储，和普通的 `vector<T>` 完全不同；或者你写了一个通用的比较函数，但对 `const char*` 不能用 `>` 来比较（那比较的是地址而不是字符串内容）。

**模板特例化**就是为这种"通用之中的特殊"而存在的。它让你为一个模板的特定类型（或条件）提供专属的实现。

------

**全特化**

"为一个具体的类型组合写一套完全独立的实现"——这就是全特化：

```cpp
// 通用模板
template<typename T>
class Comparator
{
public:
    bool greater(const T& a, const T& b) const { return a > b; }
};

// 针对 const char* 的全特化
template<>
class Comparator<const char*>
{
public:
    bool greater(const char* a, const char* b) const
    {
        return strcmp(a, b) > 0;     // 字符串内容比较，不是地址比较
    }
};
```

使用时：

```cpp
Comparator<int> ci;
cout << ci.greater(5, 3) << endl;     // true — 用 int 版

Comparator<const char*> cs;
cout << cs.greater("z", "a") << endl; // true — 用 const char* 版（内部是 strcmp）
```

全特化的语法：原模板的 `template<typename T>` 变为 `template<>`，类名后跟 `<具体类型>`。

你也可以特化单个函数，而不用把整个类特化——函数全特化语法是 `template<> void func<具体类型>(...)`。不过需要留意的是，**函数模板不支持偏特化**，只能全特化。如果你需要基于部分类型的函数特化，通常变通的做法是把它包装成一个类模板的成员函数（因为类模板支持偏特化）。

------

**偏特化**

有时候你不想特化到某个具体的类型，而是特化到"模式匹配"——比如"如果 T 是一个指针"，或者"如果两个类型相同"。这叫**偏特化**：

```cpp
// 通用版本
template<typename T>
class Container { /* ... */ };

// 针对指针的偏特化
template<typename T>
class Container<T*>    // T 被推导为指针底层类型
{
    // 适用于任何指针类型的实现
    // 比如可以在这里提供额外安全性检查、或对指针元素做特殊处理
};

// 针对两个相同类型的偏特化
template<typename T>
class Pair<T, T>
{
    // 两边类型相同时的特殊实现
};
```

标准库就是靠偏特化来实现很多高级特质的——比如 `std::is_pointer` 判断一个类型是否是指针，就是通过偏特化 + 继承 bool_constant 来吐出 true/false 的。

偏特化在泛型库开发中用得特别多。打个比方：你写了一个"所有智能指针的基础类"，在通用版本中走通常逻辑，在 `is_trivially_copyable<T>` 版本中走 memcpy 优化逻辑——这就是偏特化的一种典形场景。

------

**变量模板的特化（C++14）**

模板不仅可以是函数和类，也可以是变量（C++14 起）。自然也能特化：

```cpp
template<typename T>
constexpr T pi = T(3.14159265358979);

template<>
constexpr float pi<float> = 3.14f;      // 对 float 精度不需要太高

cout << pi<double> << endl;   // 3.14159265358979
cout << pi<float> << endl;    // 3.14
```

------

**SFINAE 与 enable_if（一个简短的预看）**

你会看到有些代码里有一堆 `typename = enable_if_t<...>`。这叫 **SFINAE**（Substitution Failure Is Not An Error）：编译器尝试推导模板参数时，如果某个候选不匹配，不报错——直接跳过，找下一个匹配的。

这套机制主要用于实现"类型约束"：比如这个函数模板只接受整数类型，不接受浮点。在 C++20 引入 Concepts 之后，很多场景可以用更清晰的 `requires` 子句代替 SFINAE。但 SFINAE 在 C++11~C++17 的现有海量代码中依然铺天盖地，识别并理解它的写法是阅读 STL 源码的基本功。

特例化+SFINAE 组合在一起，构成了 C++ 模板超图——你可以针对"所有整数类型"、"所有可复制类型"、"所有满足某某运算的类型"提供不同的实现，这一切都在编译期完成，没有运行时开销。
