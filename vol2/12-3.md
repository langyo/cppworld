# 原地不动 · 深入理解移动操作

第九章我们学了右值引用和 `std::move`。到了模板章节，我们把移动的底层逻辑再往下挖一层——`move` 和 `forward` 到底区别在哪、`noexcept` 在什么时候是命、以及编译器在背后帮你做了哪些优化。

------

## move vs forward：别再混淆

先给出最直接的区别：

- `std::move(x)` = **"请无条件把 x 当垃圾处理"**——把 x 转成右值，之后 x 的内容可能被掏空，你不应再依赖它。
- `std::forward<T>(x)` = **"请保持 x 本来的身份"**——如果外面把 x 传进来时是左值就当左值传，是右值就当右值传。只在模板的通用引用场景里用。

一个直观的例子让你看清楚它们的差别：

```cpp
#include <iostream>
#include <utility>

// 两个重载，让你看清参数到底是什么身份来的
void sink(int& x)  { cout << "收到左值: " << x << "\n"; }
void sink(int&& x) { cout << "收到右值: " << x << "\n"; }

template<typename T>
void relay(T&& arg)
{
    // arg 在这里本身是个左值——它有自己的名字、可以取地址
    sink(arg);                       // 永远走左值版本
    sink(move(arg));            // 永远走右值版本（arg 被掏空）
    sink(forward<T>(arg));      // 还原原始身份——外面是左就左，外面是右就右
}

int main()
{
    int a = 42;
    cout << "传左值:\n";
    relay(a);          // T = int&, forward 走左值

    cout << "传右值:\n";
    relay(100);        // T = int,   forward 走右值
}
```

输出：

```
传左值:
收到左值: 42
收到右值: 42
收到左值: 42
传右值:
收到左值: 100
收到右值: 100
收到右值: 100
```

`forward` 把第三行和第六行的行为区分开了——外面如果是右值进来的，`forward` 就把右值身份还给它，让下游的 `sink` 走右值重载。

核心记忆法：你是"制造者"（明确要移交所有权）→ `move`；你是"传递者"（不知道外面怎么办的，忠实转交）→ `forward`。

## 自己写一个移动感知的资源类

讲原理不如动手。下面是一个管理原始 `char*` 缓冲区的类，我们把它的拷贝构造、拷贝赋值、移动构造、移动赋值的四种行为都实现出来，看它们在不同场景下怎么被调用：

```cpp
#include <iostream>
#include <cstring>
#include <utility>

class Buffer
{
private:
    char* data;
    size_t size;

public:
    // 普通构造
    Buffer(const char* s) : size(strlen(s) + 1), data(new char[size])
    {
        memcpy(data, s, size);
        cout << "构造: \"" << data << "\"\n";
    }

    // 拷贝构造
    Buffer(const Buffer& other) : size(other.size), data(new char[size])
    {
        memcpy(data, other.data, size);
        cout << "拷贝构造: \"" << data << "\"\n";
    }

    // 移动构造（noexcept 告诉编译器这个不抛异常）
    Buffer(Buffer&& other) noexcept
        : data(other.data), size(other.size)
    {
        other.data = nullptr;     // 必须把别人掏空
        other.size = 0;
        cout << "移动构造\n";
    }

    // 拷贝赋值
    Buffer& operator=(const Buffer& other)
    {
        if (this != &other)
        {
            delete[] data;
            size = other.size;
            data = new char[size];
            memcpy(data, other.data, size);
        }
        cout << "拷贝赋值\n";
        return *this;
    }

    // 移动赋值
    Buffer& operator=(Buffer&& other) noexcept
    {
        if (this != &other)
        {
            delete[] data;              // 先释放自己原有的
            data = other.data;          // 接管别人的
            size = other.size;
            other.data = nullptr;       // 把别人掏空
            other.size = 0;
        }
        cout << "移动赋值\n";
        return *this;
    }

    ~Buffer()
    {
        delete[] data;
    }

    const char* c_str() const { return data ? data : "(empty)"; }
};
```

现在拿它做几个测试——看看编译器什么时候选拷贝、什么时候选移动：

```cpp
int main()
{
    Buffer b1("Hello World");

    Buffer b2 = b1;                    // 拷贝构造——b1 是左值
    Buffer b3 = move(b1);         // 移动构造——move 把 b1 变成右值
    cout << "b1 被移动后: " << b1.c_str() << "\n";  // (empty)

    Buffer b4("Temp");
    b4 = b2;                           // 拷贝赋值——b2 是左值
    b4 = Buffer("Anonymous");          // 移动赋值——右侧是临时对象
}
```

输出：

```
构造: "Hello World"
拷贝构造: "Hello World"
移动构造
b1 被移动后: (empty)
构造: "Temp"
拷贝赋值
构造: "Anonymous"
移动赋值
```

第五行 `b4 = Buffer("Anonymous")` 最值得细看——先构造一个匿名 `Buffer`，然后因为是右值，赋值时自动走移动赋值，最后匿名对象析构——全程没有一次不必要的字节拷贝。

## noexcept 为什么重要

标准库容器（如 `vector`）在扩容时有两种搬迁元素的方式：移动或拷贝。如果元素的移动构造函数标记了 `noexcept`，`vector` 就会优先选移动（快）；否则为了异常安全退回到拷贝（慢）。

你可以在自己的 `Buffer` 上验证这一点——把移动构造的 `noexcept` 去掉，然后用一个 `vector<Buffer>` 做多次 `push_back`，观察扩容时的输出变化（移动变拷贝）。

一条简洁规则：**凡是你的移动构造和移动赋值不需要分配资源、只交换指针的，都标上 `noexcept`。** 这是一个强承诺，`vector`、`deque` 等标准容器的行为直接依赖于它。

## RVO：编译器比 move 更聪明

返回值优化（RVO）是 C++ 标准中少数特批的优化——即使有可观察的副作用（如构造函数里的 `cout`），编译器也被允许省略拷贝/移动。

```cpp
Buffer createGreeting()
{
    Buffer result("Hello");
    // ……对 result 做其他操作……
    return result;    // 编译器直接构造到调用者的变量空间，零次额外构造
}

int main()
{
    Buffer g = createGreeting();   // 只输出一行 "构造: Hello"
}
```

**永远不要写 `return std::move(result);`**——`move` 把 `result` 变成了引用，破坏了 RVO 的前提条件，反而逼着编译器去做一次移动。让它安安静静返回局部变量，编译器自己知道该怎么做最划算。
