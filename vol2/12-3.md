# 原地不动 · 深入理解移动操作

第九章讲右值引用和移动语义时，我们集中在"大容器返回不死拷贝"上。现在到了模板章节，我们再回来挖得深一点——`std::move` 和 `std::forward` 到底有哪门子区别？什么场景下必须用哪个？`noexcept` 在这里到底起什么作用？

------

**move vs forward：不要再混淆了**

- `std::move(x)` = "无条件把 x 变成右值"。在这之后，x 的资源可能会被别人偷走，你最好不要再碰 x 的内容。
- `std::forward<T>(x)` = "按 T 的原始值类别把 x 转成相应的左值或右值"。只用在转发引用 `T&&` 场景中，用于还原参数的本来身份。

一条最容易记的准则：**如果你是"制造者"（明确想把资源转移掉），用 `move`；如果你只是"传递者"（你不决定资源的去向，只负责原样转手），用 `forward`。**

在模板的世界里，`forward` 的出现远比 `move` 多。因为模板函数通常是"传递者"——它不知道也不应该替调用者做决定，它只需要忠实地把参数交到下家手上。

------

**noexcept 在移动之中的分量**

你看标准库的移动构造函数总跟个 `noexcept` 修饰：

```cpp
vector(vector&& other) noexcept;
```

`noexcept` 是告诉编译器："这个移动操作绝不抛异常。"这有什么意义？

假设 vector 扩容的时候要搬移现有元素到一块更大的内存区。如果搬移过程中会抛异常，而部分元素已搬过去却卡在中间出事了，这时旧区已经被部分掏空，新地方又没搬完——这是个恢复不了的局面。

所以标准库的做法是：**如果元素移动操作标注了 noexcept，就用移动搬元素（高效）；否则宁可退回到拷贝，保证异常安全。** 拷贝即使半路出问题，旧区还是完整的——因为拷贝不破坏源对象。

因此，**你手写的移动构造函数和移动赋值运算符，除非有明显会抛异常的逻辑，都应该加上 `noexcept`。**

如何判断？如果你的移动构造函数只是互换几个指针（这都是不会抛异常的操作），那就标 `noexcept`。如果它还要分配内存、可能抛 bad_alloc，那就不能标。

------

**move_if_noexcept：编译器替你做判断**

标准库还有一个叫 `move_if_noexcept` 的工具：如果对象可移动且移动是 noexcept 的，那它做移动，否则退化为拷贝。`vector` 在扩容搬迁时就用来在元素身上做这个判断。

在实际代码中直接使用 `move_if_noexcept` 的场景有限，但理解它的作用能帮你搞懂 vector 扩容的底层逻辑。

------

**移动操作后，源对象的状态**

规范：**被移动后的对象处于一个"合法但未指定"的状态。** 你必须还能执行以下操作：

- 销毁它。
- 给它赋一个新值。
- 调用它所有没有前提条件的成员函数（如 `size()`、`empty()`）。

你不应该假设它还有原来的内容——说不定 `size()` 变成 0 了，说不定内容全部被清空了——但总之它的状态是一个"可以被安全新赋值或销毁"的合法状态。

这是库交互时的约定，也是自己写移动操作要求遵守的规矩。移动完老对象后如果不把它放进一个确定的新状态，后续再使用它就是不安全的。

------

**Return Value Optimization（RVO）**

最后说一个与移动密切相关的编译优化——RVO（返回值优化）。

```cpp
vector<int> create()
{
    vector<int> result;
    // 往 result 里塞一堆东西
    return result;   // 编译器不会拷贝，直接原地构造到调用者的变量空间里
}
```

你不需要写 `return move(result);`。写 `move` 反而会**阻止** RVO——因为 `move(result)` 返回的是一个引用类型，不再是待返回的局部变量本身，编译器就无法做原地构造了。一条简单规则：**return 局部对象时，就让编译器自己决定，别擅自加 move。**
