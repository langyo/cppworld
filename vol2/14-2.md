# 安全部门 · 异常处理库

在第四章我们初次见识了 `try-catch-throw` 三个关键字。但是——当时我们扔的都是随手写的字符串。真正工程中的异常处理远不止这个层次。

C++ 标准库在 `<stdexcept>` 中定义了**异常类体系**——一个有继承关系的异常族，让你可以按"大类"甚至"大类的基类"来捕获异常。

------

**异常类的家族树**

```
exception                       (所有标准异常的基类)
├── logic_error                 (逻辑错误——一般可提前预见)
│   ├── invalid_argument        (参数无效)
│   ├── out_of_range            (越界访问)
│   ├── length_error            (长度超出限制)
│   └── domain_error            (数学上的定义域错误)
├── runtime_error               (运行时错误——执行中才暴露)
│   ├── range_error             (区间错误)
│   ├── overflow_error          (算术溢出)
│   └── underflow_error         (算术下溢)
└── bad_alloc                   (内存分配失败——在 <new> 头文件中)
```

----- 

**每个异常都带有 `what()` 成员**

```cpp
try
{
    throw out_of_range("索引超出向量范围！");
}
catch (const exception& e)       // 基类引用——能捕获所有标准异常
{
    cout << "异常信息: " << e.what() << endl;
}
```

`e.what()` 返回一个 C 风格字符串，内容是抛出时给的描述。用它来记日志、显示给用户都非常方便。

------

**捕获的顺序很重要**

`catch` 块按写下来的顺序尝试匹配。如果把 `exception` 基类写在最前面，后面更具体的 `invalid_argument` 就永远不会被触发——因为所有派生类都"也满足"基类的匹配条件。

```cpp
try
{
    // ...
}
catch (const invalid_argument& e) { /* A */ }  // 具体类型在前
catch (const runtime_error& e)   { /* B */ }
catch (const exception& e)       { /* C */ }    // 基类在最后当兜底
```

如果把这段的 C 移到第一行，A 和 B 就死代码了——编译器通常还会给个警告提醒你。

------

**自定义异常类**

继承 `exception` 或它的子类来写自己的异常类型：

```cpp
class ConfigError : public runtime_error
{
public:
    explicit ConfigError(const string& msg) : runtime_error(msg) {}
};

// 抛出
throw ConfigError("配置文件格式错误");
```

继承现有异常基类让你的异常自动融入标准库的 catch 体系——如果调用者只是 `catch (const exception&)`，你的异常也能被接住。

------

**noexcept：承诺不抛异常**

`noexcept` 说明符可以告诉编译器——"这个函数绝不会抛异常"：

```cpp
int safeAdd(int a, int b) noexcept { return a + b; }
```

如果 noexcept 函数里还是抛了异常而没有人捕获，程序会直接调 `std::terminate`——连 unwind 都没有，干净利落地结束。这是故意设计的激进行为：你承诺了"不抛异常就不抛"，编译器有权据此优化代码（比如省略为异常准备的栈展开信息）。

`noexcept` 在移动构造函数、swap、析构函数上非常关键——这些地方如果抛异常，往往意味着整个程序的一致性无法保证，所以你更该标记 `noexcept` 并保证不抛。

------

**noexcept 的运算符形态**

`noexcept` 也可以当"查询符"用——编译期检查一个表达式是否标注了 noexcept：

```cpp
cout << noexcept(safeAdd(1, 2)) << endl;         // true
cout << noexcept(throw out_of_range("")) << endl; // false
```

这常用于模板元编程中做编译期能力检测——比如某个容器代码中，如果元素类型标注了移动 noexcept，就用移动搬元素，否则退化为拷贝搬。

------

异常处理在 C++ 中虽然不像 Java 那样被强制检查，但它仍是正确的错误汇报手段之一。今天的 C++ 实践中，库代码多用异常报错（因为返回值的错误通道在使用者不检查时很容易漏掉），但性能内部要避免抛异常（用断言、合约、返回值判断等手段替代）。了解标准的异常类体系，让自己的异常继承进去，是让代码像标准库一样有范儿的必由之路。
