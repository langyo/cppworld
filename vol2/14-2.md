# 安全部门 · 异常处理库

第四章我们学了 `try-catch-throw` 的基本语法。但真实项目中不能靠随手扔字符串来做异常处理——一来没法区分错误类型，二来随手写的字符串容易在代码量爆炸后失去统一性。

C++ 标准库在 `<stdexcept>` 中定义了一整套**有继承关系的异常类族**，把你的异常处理从"随便喊句话"升级到了"分级报警的三层消防系统"。

------

## 异常类的家族树

```
std::exception                    （所有标准异常的根）
├── std::logic_error              （逻辑错误——理论上能提前发现）
│   ├── std::invalid_argument    （参数传错了）
│   ├── std::out_of_range        （下标越界）
│   ├── std::length_error        （长度超过容器 max_size）
│   └── std::domain_error        （数学上超出定义域）
├── std::runtime_error            （运行中才能发现的错误）
│   ├── std::range_error          （区间错误）
│   ├── std::overflow_error       （算术上溢）
│   └── std::underflow_error      （算术下溢）
└── std::bad_alloc                （new 分配失败，在 <new> 中）
```

------

## 建立在基类之上的捕获——按大类收网

所有标准异常都有一个 `what()` 成员函数，返回 C 风格字符串描述。因此你可以在任何一个层级选择截住：

```cpp
#include <iostream>
#include <stdexcept>
#include <vector>

int main()
{
    try
    {
        std::vector<int> v(5);
        v.at(10) = 100;                     // 抛 out_of_range
    }
    catch (const std::out_of_range& e)      // 具体——先截
    {
        std::cerr << "越界: " << e.what() << "\n";
    }
    catch (const std::logic_error& e)       // 中类——没被上面截住的 logic_error 落在这
    {
        std::cerr << "逻辑错误: " << e.what() << "\n";
    }
    catch (const std::exception& e)         // 基类兜底——截住所有漏网的标准异常
    {
        std::cerr << "未知异常: " << e.what() << "\n";
    }
}
```

**捕获顺序不能倒**——如果把 `exception` 写在最前面，后面几十个派生类全都不会被执行（因为派生类也是基类，前面的 catch 就截住了）。编译器通常会对此给一个警告。

## 写自己的异常类型

继承一个标准基类来派生你业务层的异常，让它自动融入上述 catch 链：

```cpp
#include <iostream>
#include <stdexcept>

class ConfigError : public std::runtime_error
{
public:
    explicit ConfigError(const std::string& msg)
        : std::runtime_error(msg) {}
};

class NetworkError : public std::runtime_error
{
private:
    int httpCode;
public:
    NetworkError(int code, const std::string& msg)
        : std::runtime_error(msg), httpCode(code) {}

    int getHttpCode() const { return httpCode; }
};

void loadConfig(const std::string& path)
{
    if (path.empty())
        throw ConfigError("配置文件路径为空");

    // 模拟解析失败
    throw ConfigError("配置文件中存在语法错误");
}

int main()
{
    try
    {
        loadConfig("");
    }
    catch (const ConfigError& e)
    {
        std::cerr << "配置异常: " << e.what() << "\n";
        // 可以在这里做针对配置错误的重试或默认值回退
    }
    catch (const std::runtime_error& e)   // 包括 ConfigError 在内的所有 runtime 分支
    {
        std::cerr << "运行时错误: " << e.what() << "\n";
    }
}
```

你派生的 `ConfigError` 继承 `runtime_error`，就天然继承了 `what()`——你交给基类构造函数的 `msg` 字符串会被存起来，任何人沿着 `const exception&` 往下截也能读到。

## noexcept：你承诺不抛、编译器帮你优化

很多关键位置（移动构造函数、析构函数、swap）必须承诺不抛异常。在函数声明后面加上 `noexcept`：

```cpp
class Buffer
{
public:
    Buffer(Buffer&& other) noexcept { /* 只交换指针，不抛异常 */ }

    int safeAdd(int a, int b) noexcept { return a + b; }

    // 如果这个函数真的抛了异常但没有被内部 catch 接住，程序直接 std::terminate
};
```

`noexcept` 不仅仅是文档式承诺——标准库的容器会检查搬移操作是否有 `noexcept` 来决定扩容时用拷贝还是搬移。写了 `noexcept`，`vector` 就敢在扩容时对你的对象执行移动（高效）；不写或不能保证则不信任，退回到拷贝。
