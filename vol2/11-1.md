# 从原子到宇宙 · 面向对象概述

第七章我们学会了类——把相关数据和函数捆在一起，形成一个新的类型。这本身就是一种进步：原来散落一地的数据有了归宿，代码也更易读。

但如果每种类型之间的相似之处都只能拿反复的代码来表达，那"新类型"这个概念本身还不够劲。想想看：

- 一个 `Bird` 和一只 `Dog` 都是动物，它们都会叫，但叫声不一样。
- 一个 `Circle` 和一个 `Rectangle` 都是图形，它们都有面积，但计算方法不同。
- 一个 `EmailNotification` 和一个 `SMSNotification` 都是通知，发送方式各异。

理想情况下，我们希望代码能这样表达："鸟是动物，狗也是动物，他们都会叫自己的叫声"——而不是为每种动物把同样的流程抄一遍。

这就是**面向对象编程**的核心意义：让类型之间产生关系，让相同的东西只写一次，让不同的东西自己做主。

------

面向对象建立在三个核心支柱上：

- **封装**：把数据和操作藏在一起（第七章已经会了）。
- **继承**：新的类型从已有的类型中"继承"已有的数据和函数，再添加自己的特有部分。
- **多态**：用基类的接口统一操作不同的派生类，实际运行时会调派生类自己的版本。

我们一步步来。先把一个很朴素的问题摆在面前，顺着解决它，把面向对象的思想自然推导出来。

------

假设我们在写一个《Minecraft》式的游戏。有两种方块——土方块和石头方块。它们的 id 不同，被破坏时的行为也不同：土块直接碎了，石头爆炸。

**方案一：各写各的类。** 两个类有大量重复的代码（id、耐久值、挖掘方法），每次写新方块都要抄一遍，改一个地方要改 N 个文件——没法维护。

**方案二：提取公共部分为一个基类，两种方块继承它。** 公共的行为只写一次，各自独特的行为由子类自己覆盖。

方案二就是面向对象的味道。代码长这样：

```cpp
class Block            // 基类——所有方块的通用属性
{
protected:
    int id;
    int hardness;
public:
    Block(int id, int hardness) : id(id), hardness(hardness) {}
    virtual void onDestroyed() { cout << "方块碎了" << endl; }  // virtual 是关键
};

class DirtBlock : public Block    // 土块继承自方块
{
public:
    DirtBlock() : Block(1, 5) {}     // id=1，硬度 5
    void onDestroyed() override      // 覆盖基类行为
    {
        cout << "泥土散落一地" << endl;
    }
};

class StoneBlock : public Block
{
public:
    StoneBlock() : Block(2, 50) {}   // id=2，硬度 50
    void onDestroyed() override
    {
        cout << "石头碎裂飞溅" << endl;
    }
};
```

`virtual` 关键字告诉编译器："这个函数将来可能在子类中被覆盖。" `override` 则是对编译器明确声明"我就是在覆盖基类的那个函数，不是另起炉灶写了个重名函数"。

如果没有 `virtual`，哪怕你在子类写了一个同名函数，通过基类指针/引用调用时，编译器只会调用基类的版本——因为编译器默认按"静态类型"来选择，这就是所谓的**静态绑定 vs 动态绑定**。

---

现在，我们能不能用一种统一的方式跟不同类型的方块对话？

```cpp
void breakBlock(Block& b)
{
    cout << "正在破坏方块..." << endl;
    b.onDestroyed();
}

DirtBlock dirt;
StoneBlock stone;

breakBlock(dirt);    // 正在破坏方块...  泥土散落一地
breakBlock(stone);   // 正在破坏方块...  石头碎裂飞溅
```

函数 `breakBlock` 只知道自己接收的是 `Block&`，它根本不了解当前的方块是土还是石。但是，正是因为 `onDestroyed` 声明了 `virtual`，运行时才会找"真实的类型"去调用对应版本。这就是**多态**——一个接口，多种行为。

如果把 `virtual` 去掉重试一遍——你会发现不管传什么，`breakBlock` 都只会打"方块碎了"这一行。这就是缺乏多态时，静态类型（`Block&`）取代动态类型（实际传入的对象是 `DirtBlock` 还是 `StoneBlock`）的后果。

------

**用多态管理对象集合**

日常写代码经常碰到这样的需求：把一堆不同子类的对象放在一个集合里管理，然后统一操作。

```cpp
vector<unique_ptr<Block>> blocks;         // 存索引而非对象
blocks.push_back(make_unique<DirtBlock>());
blocks.push_back(make_unique<StoneBlock>());
blocks.push_back(make_unique<DirtBlock>());

for (auto& b : blocks)
    b->onDestroyed();                     // 批量破坏
```

`unique_ptr<Block>` 存的是 DirtBlock 和 StoneBlock——因为基类指针可以指向派生类对象，多态才得以运作。

这里一定要用指针或引用而不是值——如果你写 `vector<Block>` 并试图往里塞 `DirtBlock`，它会默默变成 `Block`（切片了，Dirt 的独有部分被切掉）。这是 C++ 中 OOP 新手必踩的坑。

------

**基类应该定义虚析构函数**

在上面的 `blocks` vector 中，当 `unique_ptr<Block>` 析构时，它会调用 `Block` 的析构函数。如果 `Block` 的析构函数不是虚的，只会调用 `Block` 自己的析构，派生类里额外持有的资源就没人清理了——泄漏。

修正办法：

```cpp
class Block
{
public:
    virtual ~Block() = default;   // 虚析构，保证派生类的清理被调用
    // ...
};
```

任何一个可能被当作基类的类，都应该把析构函数标为 `virtual`。这是 C++ 面向对象中最基本的铁律。

---

我们刚刚经历了面向对象编程的一个微缩之旅——把相同代码升到基类，把差异留在派生类，用 `virtual` 和指针开启多态。接下来几节，我们深入这些概念的细节——继承模型、虚函数表、访问控制、以及编译器替你生成的合成函数在 OOP 中的行为。
