# 从原子到宇宙 · 面向对象概述

第七章我们学会了类——把数据和操作它的函数捆成一个新的类型。这已经让代码比原来散落一地的变量强太多了。

但真实世界里类型之间是有关系的。"鸟是动物，狗也是动物。它们都会叫，但声不一样。"如果用传统的朴素类，每一种动物都得把"吃饭、睡觉、叫"这些代码从头写一遍——重复且难维护。

**面向对象编程**就是把这种"类型间的关系"写进代码里的系统方法。它靠在三个支柱之上：

- **封装**（第七章已会）——数据和操作藏在一起。
- **继承**——新类型从旧类型"继承"已有的特性，再添加自己的。
- **多态**——用基类的接口统一调度派生类，实际运行时会执行派生类自己的版本。

------

## 从实际问题推导出面向对象

假设你在写一个《Minecraft》风格的方块破坏系统。游戏里有土块、石块、木块。每种块的 id 不同，被敲碎时的表现也不同：土块散了，石块碎裂飞溅，木块掉落树苗。

**方案一：各写各的。** 三种方块三个互不相干的类。破坏方块的代码得这样写：

```cpp
if (type == DIRT)    DirtBlock::onDestroyed();
if (type == STONE)   StoneBlock::onDestroyed();
if (type == WOOD)    WoodBlock::onDestroyed();
```

每加一种新方块，这份 if-else 链条就长一截。改了某个方块的行为，可能有三四个地方的判断都得跟着改。

**方案二：提炼基类，让各个方块继承它。** 破坏方块的代码变成一行——不管什么方块，调同一个方法：

```cpp
block->onDestroyed();    // 管你土块石块木块，自己搞定
```

代码长这样：

```cpp
class Block
{
public:
    virtual ~Block() = default;

    virtual void onDestroyed() const       // virtual = 子类可以覆盖
    {
        std::cout << "方块碎了\n";
    }
};

class DirtBlock : public Block
{
public:
    void onDestroyed() const override      // override = 我就是要覆盖基类的这个函数
    {
        std::cout << "泥土散落一地\n";
    }
};

class StoneBlock : public Block
{
public:
    void onDestroyed() const override
    {
        std::cout << "石头碎裂飞溅\n";
    }
};
```

`virtual` 是开启多态的钥匙——它告诉编译器："这个函数在调用时请查一下对象的真实类型，去找最贴切的版本来执行。"没有 `virtual`，即使你在子类写了同名的函数，通过基类指针/引用调的时候也只会执行基类的版本。

`override` 是可选的但强烈建议写上——它让编译器帮你核对"基类是不是真的有这么个虚函数"。如果哪天基类改了名字而你忘了同步改子类，`override` 会立刻报编译错误，而不是静默地变成不相干的新函数。

## 外部代码怎样享用多态

```cpp
// 这个函数只知道自己收到的是 Block&，完全不需要管是土是石
void breakBlock(const Block& block)
{
    std::cout << "正在破坏方块...\n";
    block.onDestroyed();           // 虚函数——运行时找真实类型分派
}

int main()
{
    DirtBlock dirt;
    StoneBlock stone;

    breakBlock(dirt);    // 正在破坏方块...  泥土散落一地
    breakBlock(stone);   // 正在破坏方块...  石头碎裂飞溅
}
```

把它放进容器里批量处理：

```cpp
#include <vector>
#include <memory>

int main()
{
    std::vector<std::unique_ptr<Block>> world;    // 存指针而不是对象本体
    world.push_back(std::make_unique<DirtBlock>());
    world.push_back(std::make_unique<StoneBlock>());
    world.push_back(std::make_unique<DirtBlock>());

    for (const auto& block : world)
        block->onDestroyed();    // 一次性破坏所有方块，各用各的效果
}
```

这里必须用 `unique_ptr<Block>` 而不是直接 `vector<Block>`——如果你写 `vector<Block>` 并往里塞 `DirtBlock`，对象会被**切片**（slicing）：`DirtBlock` 中独有的部分被截掉，只剩下 `Block` 那部分。多态必须通过指针或引用才能生效。

------

## 基类不要忘了虚析构

上面的 `Block` 基类中，析构函数写的是 `virtual ~Block() = default;`。这不是可选项——任何一个可能被作为基类的类，**都**应该有一个虚析构函数。

为什么？如果不用虚析构：

```cpp
Block* ptr = new DirtBlock();
delete ptr;     // 只调 Block 的析构，DirtBlock 里额外分配的资源直接泄漏
```

因为 `ptr` 静态类型是 `Block*`，编译器老老实实按这个类型去调析构函数。`Block` 的析构不是虚的，就不会触发 `DirtBlock` 的析构。把析构标上 `virtual` 之后，`delete` 基类指针时会通过虚表找到正确析构函数——从最底层的派生类一层层往上调，干干净净。

------

## 接口与抽象基类

有时候基类本身就是"概念的占位符"——它只定义一组派生类必须遵守的契约，自己不去实现任何具体行为。这种基类含有**纯虚函数**：

```cpp
class IDrawable
{
public:
    virtual void draw() const = 0;    // "= 0" 表示纯虚——只声明，没有实现
    virtual ~IDrawable() = default;
};
```

任何包含纯虚函数的类都是**抽象类**，你无法直接创建它的对象——`IDrawable d;` 编译报错。派生类要实现这个接口，必须提供所有纯虚函数的覆盖版本。

```cpp
class HealthBar : public IDrawable
{
public:
    void draw() const override
    {
        std::cout << "绘制血条\n";
    }
};

class Minimap : public IDrawable
{
public:
    void draw() const override
    {
        std::cout << "绘制小地图\n";
    }
};
```

使用方式和前面一模一样——通过基类指针/引用操作，完全不知道也不需要知道具体是什么：

```cpp
std::vector<std::unique_ptr<IDrawable>> hudElements;
hudElements.push_back(std::make_unique<HealthBar>());
hudElements.push_back(std::make_unique<Minimap>());

for (auto& elem : hudElements)
    elem->draw();
```

C++ 没有 Java 那样的 `interface` 关键字，但上面的纯虚抽象类就是等价物——它能允许多重继承（一个类实现多个接口）而不引入"钻石问题"（因为接口没有数据成员）。

------

面向对象的真正威力不在于语法，而在于当你把"同"升到基类、"异"留在派生类之后，整个系统的心智负担骤然降低——新增一种方块只需写一个新 class，加几个新的覆盖函数，其它地方一行都不用改。这种可扩展性，是那之前的任何平坦组织方式都做不到的。
