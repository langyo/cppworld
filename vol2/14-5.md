# 俄罗斯套娃 · 局部类与嵌套类

类不一定非得放在全局作用域或命名空间里。它可以塞在另一个类里面，甚至可以塞在一个函数体内。这种"套娃"设计在特定的封装场景里很管用——比如只给宿主类使用的辅助类型、或者在大函数内部定义临时数据结构。

------

**嵌套类：一个类套在另一个类里面**

```cpp
class BinaryTree
{
private:
    struct Node           // 这是嵌套类——对 BinaryTree 外世界完全隐藏
    {
        int value;
        Node* left;
        Node* right;
    };

    Node* root;

public:
    // 外部代码完全看不到 Node 的存在，只能通过 BinaryTree 间接操作
};
```

嵌套类 `Node` 完全属于 `BinaryTree` 的私有部分——外界的代码既不知道 Node 的名字也用不了 Node 的类型。这是信息隐藏的终极方式：连类型本身你都看不见，更不要说尝试构造。

如果外部代码仍然需要拿到嵌套类的对象，但又不能暴露其内部成员，常规手法是把它声明为 public 但只向外提供接口（如一个纯粹的虚基类）——这种模式在标准库的一些迭代器设计里出现过，主要就是为了保护迭代器内部状态不被外部修改。

------

**局部类：塞在函数里面的类**

```cpp
void process()
{
    class TempHelper      // 局部类——只在本函数内部有效
    {
    public:
        int compute() { return 42; }
    };

    TempHelper h;
    cout << h.compute() << endl;
}
```

局部类的成员函数必须在类内部内完整定义（因为类本身不在命名作用域里，没办法在外面定义）。局部类不能有静态成员，也不能访问函数外的局部变量。用处比较有限，但也有——比如在算法中需要一个只在当前函数里使用的临时数据结构，且这个结构简单到不值得拖到文件域。

------

**嵌套类与访问权限**

嵌套类毕竟在宿主类的命名空间下，它能否看到宿主类的 `private` 成员？

不能。**嵌套类和宿主类只是名字上的隶属关系，访问权限上没有特殊待遇**——它就是宿主类友元之外的一个普通类。反过来，宿主类也不能直接访问嵌套类的 private 成员（除非嵌套类把宿主类声明为友元）。

```cpp
class Outer
{
private:
    int secret;

    class Inner
    {
    public:
        void peek(Outer& o)
        {
            // cout << o.secret;    // ❌ Inner 不是 Outer 的友元，拒访
        }
    };
};
```

如果嵌套类必须访问宿主类的 private 成员，需要进行友元声明——`friend class Inner;`。需要提醒的是，过多的友元关系会快速破坏封装性，先想清楚是不是把这两个类在逻辑上真的该放在一起。

------

嵌套类和局部类是颇偏门的工具——它们的出现一般是因为你要保护一个"类型名字"不被外部滥用，或是因为某个辅助逻辑只在极窄的作用域内出现而不值得全局可见。当你在某个函数或类里面写一小段复杂的逻辑而需要一个自带的类型时，想想这里的套娃——也许正是清爽解耦的手段。
