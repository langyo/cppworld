# 瞬移术 · 右值引用与对象移动

来看一段日常代码：

```cpp
vector<int> createBigVector()
{
    vector<int> result;
    // 往 result 里塞了很多很多数据……
    return result;
}

int main()
{
    vector<int> v = createBigVector();
    // 问：result 是怎么交到 v 手上的？
}
```

直觉上，`result` 是函数内部的局部变量，函数结束后它应该被销毁。那它的数据怎么到了 `v` 手里？难道是被整个拷贝了一份？

在 **C++11 之前**，确实是拷贝——先复制每一块数据，再销毁源数据，白费功夫。

**C++11 之后**，标准库和编译器跟商量好了似的，用了**移动**（Move）语义——不拷贝数据，而是把数据的所有权直接"瞬移"过去。老对象只剩一个空壳，新对象接管一切。移动同样也会让老对象销毁，但数据不用"复制一遍再删一遍"了。

------

**左值与右值**

要理解移动，我们先得分清楚**左值**和**右值**。

最简单的辨别法：

- **能取地址**的表达式 → **左值**。
- **不能取地址**的表达式 → **右值**。

```cpp
int a = 10;          // a 是左值——你可以写 &a
int b = a;           // a 在右边时也算左值（它有名有姓）
int c = a + 5;       // (a + 5) 是一个临时结果，你不能写 &(a+5)，它就是右值
int d = 42;          // 42 是字面量，当然是个右值
```

函数返回的值（非引用返回）也是右值——它没有稳定的身份，马上就消亡了。

C++11 给右值单独设计了一个引用类型：`&&`——**右值引用**。它只能绑定到右值（即将消亡的对象）：

```cpp
int&& rr = 42;           // ✅ 42 是右值
int&& rr2 = a + b;       // ✅ a+b 是右值（临时结果）
// int&& rr3 = a;        // ❌ a 是左值，不能绑到右值引用
```

与之对应，传统的 `&` 引用称为**左值引用**，它只能绑定到左值（持久的、有名有姓的变量）。

------

**移动构造与移动赋值**

标准库的容器（如 `vector`、`string`）都实现了**移动构造函数**和**移动赋值运算符**。当它们发现自己可以从一个右值（即将消亡的对象）接手资源时，就会选择移动而不是拷贝：

```cpp
vector<int> v1 = {1, 2, 3, 4, 5};
vector<int> v2 = v1;            // v1 是左值 → 拷贝构造（v1 保留）

vector<int> v3 = move(v1);      // move 把 v1 转成右值 → 移动构造
// 这时 v1 被"掏空"，变成一个空的 vector，数据全在 v3 里
```

`std::move` 的名字有点误导人——它并不是自己执行"移动"，它只是个**转换工具**：把一个左值转成右值引用。真正的移动操作发生在赋值或构造对象的那一刻，由接受右值引用的重载版本来干活。

------

**写出支持移动的类**

你的类也可以自定义移动构造函数和移动赋值运算符：

```cpp
class Buffer
{
private:
    int* data;
    size_t size;
public:
    // 普通构造
    Buffer(size_t s) : size(s), data(new int[s]) {}

    // 移动构造——"偷走"别人的资源
    Buffer(Buffer&& other) noexcept
        : data(other.data), size(other.size)
    {
        other.data = nullptr;   // 把别人的指针置空，防止 double free
        other.size = 0;
    }

    // 移动赋值
    Buffer& operator=(Buffer&& other) noexcept
    {
        if (this != &other)
        {
            delete[] data;          // 先释放自己原有的
            data = other.data;      // 接管别人的
            size = other.size;
            other.data = nullptr;   // 把别人掏空
            other.size = 0;
        }
        return *this;
    }

    ~Buffer() { delete[] data; }
};
```

要点：移动操作必须保证"掏空"源对象——把它的指针设成 `nullptr`，否则源对象的析构函数会还掉那块内存，最后两个对象都指着一块被释放的内存，炸得灰飞烟灭。这就是双重释放（double free）问题。

`noexcept` 是告诉编译器这个移动操作不会抛异常——这对标准库容器很重要，因为 vector 在扩容时如果移动操作被标注为 noexcept 就会尽量用移动而不是拷贝，否则为了"安全"会降级为拷贝。

------

**什么时候该写移动**

大多数类不需要你手写移动构造函数和移动赋值。如果你的类只包含内置类型和标准库类型（string、vector 等），编译器会自动为你生成移动操作，并且它们已经是合理的。

你只会在这些情况需要手写：
- 类内部直接管理着原始资源（裸指针、文件句柄等），像上面 `Buffer` 的例子。
- 类对某些成员有特殊的移动逻辑（不能简单地照搬编译器生成的版本）。

这和拷贝构造的法则是一样的——"三/五法则"：如果你手写了析构函数、拷贝构造函数或拷贝赋值中的任意一个，那通常需要把另外两个也显式写好，并且 C++11 以后也应顺手写移动构造和移动赋值。

------

**移动的价值**

移动语义是现代 C++ 性能优化的核心之一。它让以下场景从"慢得不能忍"变成了"快得不值一提"：

- 从函数中返回大容器（如 `vector`、`string`）。
- 往容器中塞大对象。
- swap 操作（现在都是移动交换内部指针，O(1)）。
- `unique_ptr` 的所有权转移。

一句话总结：**拷贝要复制，移动只换指针。** 能移动的时候别拷贝，让代码既干净又高效。
