# 摆脱单亲状态 · 多重继承与虚继承

前面我们看过的继承都是"一人认一个爹"——一个派生类只继承一个基类。但 C++ 允许一个类同时拥有**多个父类**，这叫**多重继承**。这个技能是一个双刃剑——用好了威力巨大，用疵了就把自己的代码送到调试地狱里夹磨。

------

**多重继承的基本语法**

```cpp
class A
{
public:
    int a;
    void show() { cout << "A" << endl; }
};

class B
{
public:
    int b;
    void show() { cout << "B" << endl; }
};

class C : public A, public B    // 继承两个基类
{
public:
    int c;
};
```

C 同时拥有 `a`、`b`、`c` 三个成员变量。但——问题来了：A 和 B 都有一个 `show()`，当你写 `c.show()` 时，编译器分不清是想调用 A 的还是 B 的。遇到同名函数，必须显式指定：

```cpp
C obj;
obj.A::show();   // 指明调用 A 的版本
obj.B::show();   // 指明调用 B 的版本
```

------

**钻石问题（菱形继承）**

多重继承中最经典的一劫。图解一下：

```
     Person
    /      \
Student   Teacher
    \      /
   Assistant
```

`Assistant` 同时继承了 `Student` 和 `Teacher`，而它们两个又都继承自 `Person`。这样一来，`Person` 的成员变量——比如 `name`——在 `Assistant` 里有两份独立的拷贝！一份来自 Student 这条链，一份来自 Teacher 这条链。你访问 `name` 必须写清楚 `Assistant::Student::name` 还是 `Assistant::Teacher::name`，否则又是二义。

显然这并不符合我们对"助理是个唯一的人"的直觉——我们要的是"只有一个 Person 的"。

------

**虚继承：解决钻石问题的利刃**

用 `virtual` 继承来确保"祖辈只存在一份"：

```cpp
class Person
{
public:
    string name;
};

class Student : virtual public Person   // 虚继承
{
public:
    int studentId;
};

class Teacher : virtual public Person
{
public:
    int teacherId;
};

class Assistant : public Student, public Teacher
{
public:
    void intro()
    {
        cout << name << endl;     // 只有一个 name 了——不再二义！
    }
};
```

`virtual` 继承告诉编译器：**沿这个继承路径，Person 只能出现一次**，即使有多条路径通过各种派生类汇聚回来，Person 最终也只保留一份。

代价是虚基类的初始化链变得非常复杂：虚基类必须由**最底层的派生类直接负责初始化**，而不是由中间层（如 Student 或 Teacher）代劳。实际写代码时你会碰到更多构造细节，建议先记住核心思想：**菱形继承用 virtual 来打破冗余拷贝。**

------

**多重继承的实际用途**

多重继承（尤其是虚继承）在日常 C++ 开发中并不常用，但有两个场合你几乎躲不开：

1. **接口类**（类似 Java 的 interface）：一个类实现多个纯虚函数接口——比如 `class MyClass : public IDrawable, public ISerializable`——这是完全合法且干净的多重继承。因为接口类没有数据成员，自然不会引发钻石问题。

2. **混合型设计**：把标准库的非虚基类（如 `std::enable_shared_from_this`）和业务类合并成一个新类——通过 public 继承业务基类和另一个"能力组件"，也很常见。

除此之外的大多数深层次多重继承（尤其是带数据成员的），都是过度设计的典型信号。C++ 社区有一条简单的共识：**如果需要多重继承，先看是不是可以把额外的基类换成类成员（组合）；如果需要虚继承，重新评估是不是继承层级本身画得太深了。**
