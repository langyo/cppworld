# 无中生有 · 初识动态内存

到现在为止，我们创建的变量统统住在**栈**上。栈上的变量就像住旅店——开了房间（进入作用域）就给你用，退房时间一到（出了作用域）自动清铺。方便得很，但有两条硬限制：

1. **大小必须在编译期就知道。** 你没法在程序跑到一半时跟栈说："嘿，我现在才知道数组得是 1000 个元素——再给我划一块呗？"
2. **寿命不能超出所在的大括号。** 函数内部的局部变量，函数一返回，它就灰飞烟灭了。

很多时候我们需要突破这两条限制——运行时由用户输入决定数组大小、或者让数据活得比创建它的函数更久。这就需要**堆**——也就是动态内存。

------

## new 与 delete：造物与回收

C++ 里用 `new` 在堆上申请空间，用 `delete` 归还：

```cpp
int main()
{
    int* p = new int;         // 在堆上分出 4 个字节，返回这片空间的地址
    *p = 42;                  // 往那写个值
    cout << *p << "\n";  // 读出来

    delete p;                 // 归还！千万别忘了这行
    return 0;
}
```

`new` 做两件事：
1. 在堆上找一块空闲的、恰好能装一个 `int` 的内存。
2. 返回这块内存的首地址。

`delete` 做反事——把地址指向的那块堆内存归还给操作系统。

如果忘了写 `delete`，那块堆内存就成了**泄漏**（leak）——程序手里的土地越来越多，但你忘了把不用的地还给操作系统。长时间运行的程序（比如服务器）会因为这个慢慢吞光所有可用内存。

## 在堆上造数组

```cpp
#include <iostream>

int main()
{
    int n;
    cout << "要多少个元素？";
    cin >> n;

    int* arr = new int[n];      // n 是运行时由用户决定——栈数组做不到！

    for (int i = 0; i < n; ++i)
        arr[i] = i * 10;

    for (int i = 0; i < n; ++i)
        cout << arr[i] << " ";

    delete[] arr;               // 数组释放用 delete[]，不是 delete！
    return 0;
}
```

`new[]` 和 `delete[]` 必须成对出现——混用会导致只释放了头一个元素而后面全部泄漏，某些编译器下甚至会直接崩溃。

## 在堆上造对象

不仅内置类型，自定义类的对象也可以在堆上构造，而且可以同时调用构造函数：

```cpp
#include <iostream>
#include <string>

class Player
{
private:
    string name;
    int hp;
public:
    Player(const string& n, int h) : name(n), hp(h)
    {
        cout << name << " 加入了游戏，血量 " << hp << "\n";
    }

    void takeDamage(int dmg)
    {
        hp -= dmg;
        cout << name << " 受到 " << dmg << " 点伤害，剩余血量 " << hp << "\n";
    }

    ~Player()
    {
        cout << name << " 离开了游戏\n";
    }
};

int main()
{
    Player* steve = new Player("Steve", 100);   // 堆上构造，同时调用构造函数
    steve->takeDamage(30);                       // 用箭头访问成员
    delete steve;                                // 析构函数在这里被调用
    return 0;
}
```

输出：

```
Steve 加入了游戏，血量 100
Steve 受到 30 点伤害，剩余血量 70
Steve 离开了游戏
```

`new Player("Steve", 100)` 在堆上分配空间、然后调用构造函数初始化——一切一气呵成。`delete` 时析构函数自动跑完，空间归还。

## 堆 vs 栈：一张表看清

| | 栈 | 堆 |
| ---- | ---- | ---- |
| 谁来管理 | 编译器自动 | 程序员手动（new/delete） |
| 大小限制 | 通常几 MB | 系统可用多少就多少 |
| 速度 | 极快（移动指针） | 较慢（查空闲块表） |
| 适合什么 | 小而短的、已知大小的 | 大的、未知大小的、活得久的 |
| 会不会泄漏 | 不可能 | 忘记 delete 就会 |

那为什么不能把什么都往堆上扔？因为堆分配比栈慢一个数量级以上，而且手动管理早晚要出错。用得少却用得对的堆内存，才不辜负 C++ 给你的这份完全控制权。

## new/delete 的问题：容易忘、容易错

看一个真实容易出错的场景：

```cpp
int* createArray(int n)
{
    int* arr = new int[n];
    // 中间可能抛异常、可能提前 return……
    return arr;
}

void process()
{
    int* data = createArray(1000);
    // 用 data 做一堆事……
    // 中间某个地方 if (error) return;   ← 直接出去了，data 没 delete！
    delete[] data;
}
```

人写的代码复杂到一定程度后，`new` 和 `delete` 在代码里天各一方，漏掉某条分支路径几乎是必然的。更别说如果在 `new` 和 `delete` 之间抛了异常——`delete` 根本就不会被执行。

这就是为什么现代 C++ 教我们优先使用**智能指针**（下一节细讲）——它们利用析构函数自动归还内存，彻底消除手动 `delete` 的遗忘风险。
