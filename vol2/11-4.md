# 遗产分配 · 访问控制与派生类作用域

面向对象的第三根柱子是封装。在继承这条路上，封装具体到了一个问题：派生类到底应该能访问基类的什么东西？外部代码再通过派生类又能看到什么？

这些规则虽然有点细，但吃过一次 `protected` 用错、甚至 `private` 继承搞乱的亏就不会忘了。

------

**访问控制回顾**

| 修饰符 | 自身类内部 | 派生类内部 | 外部代码 |
| ------ | ---------- | ---------- | -------- |
| `public` | 可以 | 可以 | 可以 |
| `protected` | 可以 | 可以 | **不可以** |
| `private` | 可以 | **不可以** | **不可以** |

`protected` 是专门为继承设计的——它让派生类能"看到"基类的一部分东西，同时仍然对外部的世界隐藏。比如一个图形基类想把顶点缓存开放给子类用于渲染加速，但不希望外部业务代码直接操作它——用 `protected` 就是这类需求的答案。

`private` 是"基类自己的小秘密"——不仅外部看不到，派生类内也完全无法访问。但如果派生类非要访问某个 private 成员，只能通过基类提供的 public 或 protected 的接口（getter/setter 等）。这符合封装的原则——只能通过可靠的门禁进入，不能破墙而入。

------

**继承方式对访问控制的影响**

```cpp
class A
{
public:
    int a;
protected:
    int b;
private:
    int c;
};

class B : public A { /* ... */ };     // public 继承
class C : protected A { /* ... */ };  // protected 继承
class D : private A { /* ... */ };    // private 继承
```

| 继承方式 | 基类 `public` → 变为 | 基类 `protected` → 变为 | 基类 `private` → |
| -------- | -------------------- | ----------------------- | ---------------- |
| `public` | public | protected | 不可访问 |
| `protected` | protected | protected | 不可访问 |
| `private` | private | private | 不可访问 |

对于 public 继承（最常见），派生类基本保持基类的访问级别——外部代码看派生类就跟看基类一样，都能看到 public 的部分。

但对于 protected 继承和 private 继承，外部代码再通过派生类就看不基类的公开成员了。这两种继承不是通常的面向对象关系的标准姿势，更多是用于非常规的重用：比如 private 继承相当于"用基类的实现但不宣告外部世界我继承了它"。它们在现代 C++ 中相当少见，这里列出来主要是为了你看旧代码时不发愣。

------

**派生类作用域中名字的查找顺序**

当你在派生类的成员函数里写了一个名字，编译器按这个顺序找：

1. 当前派生类作用域
2. 直接基类的作用域
3. 间接基类的作用域（基类的基类，往上追溯）
4. 全局作用域

如果同名变量出现在多层作用域中，内层的会掩盖外层的（跟局部变量掩盖全局变量的道理一样）。

```cpp
class Base
{
public:
    int value = 10;
};

class Derived : public Base
{
public:
    int value = 20;     // 掩盖了基类的 value

    void print() const
    {
        cout << value << endl;          // 20（派生类自己的 value）
        cout << Base::value << endl;    // 10（显式指定基类的 value）
    }
};
```

用 `Base::` 前缀显式指定要访问基类的那一份，是解决名称掩盖的标准手法。

------

**友元不会被继承**

`friend` 声明不会沿继承链往下传。"你父亲的好友，跟你没关系"。

```cpp
class Base
{
    friend class A;
};

class Derived : public Base
{
    // 类 A **不是** Derived 的友元，无法访问 Derived 新增的 private 成员
};
```

友元也不具备可传递性或对称性（A 是 B 的友元不代表 B 是 A 的友元），这些都需要各自显式声明。

------

**静态成员在继承中的表现**

静态成员只属于定义它的类。派生类可以访问它（如果访问权限允许），可以使用 `派生类::静态成员` 来通过派生类名字来拿，但实际上它仍然是基类的那个唯一副本——不会因为派生而多复制一份。

在写一个库的基类时，常见的用法是提供 protected static 工厂函数，让派生类共享初始化逻辑，但不开放给外部用户。这也是 protected 除了成员变量之外的合理使用场景。
