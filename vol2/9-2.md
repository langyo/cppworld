# 封印符文 · 智能指针

上一节我们学会了 `new` 和 `delete`。它们就像手动开关阀门——当你只有一两个阀门的时候，算不了什么；但当你的程序有成百上千条"内存管道"，谁也不能保证每一处都还记得把阀门关掉。

C++11 引入了**智能指针**来替我们管这件事。把 `new` 出来的指针往智能指针手里一交，你就再也不用手动 `delete` 了——智能指针会在合适的时机替你归还内存。

智能指针藏在 `<memory>` 头文件里。主要有两位：

| 智能指针 | 一句话概括 |
| -------- | ---------- |
| `unique_ptr` | 独占所有权——这块内存只有我一个人持有 |
| `shared_ptr` | 共享所有权——多个持有者，等最后一个持有人消失时自动释放 |

------

**unique_ptr：独占所有权**

```cpp
#include <memory>
#include <iostream>
using namespace std;

int main()
{
    // 创建一个 unique_ptr，接管 new 出来的 int
    unique_ptr<int> p(new int(42));
    // 等价于（C++14 起推荐）：
    auto p2 = make_unique<int>(42);

    cout << *p << endl;      // 通过智能指针访问内容：42

    // p 离开作用域时，delete 自动执行。你再也不用写了！

    return 0;
}
```

`unique_ptr` 的名字暗示了一个重要特点——你不能拷贝它，只能**移动**它：

```cpp
unique_ptr<int> p1 = make_unique<int>(100);
// unique_ptr<int> p2 = p1;           // ❌ 编译错误！不能拷贝
unique_ptr<int> p2 = move(p1);        // ✅ 可以移动——所有权从 p1 转移到了 p2
// 现在 p1 是空的了（nullptr），p2 持有那块内存
```

移动之后，原来的指针就成了空壳——如果还去访问它，那就是典型的空指针访问错误。move 之后的 p1 不能再用作读取，但可以重新被赋予新的 unique_ptr（比如通过另一个 move 或新 make_unique）。

这背后的设计逻辑是：一块内存只能被一个人独占。如果你想要"共享管理权"，那就是下一位——`shared_ptr`——的事。

------

**unique_ptr 管理动态数组**

```cpp
auto arr = make_unique<int[]>(10);     // 管一个长度为 10 的 int 数组

for (int i = 0; i < 10; ++i)
    arr[i] = i;                         // [] 能用

// delete[] 自动调用，不用操心
```

`make_unique<int[]>(10)` 创建的 unique_ptr 知道自己是管数组的，析构时会正确调用 `delete[]` 而不是 `delete`。

------

**shared_ptr：共享所有权**

`shared_ptr` 允许多个指针指向同一块内存。它通过一个**引用计数**来追踪——每当有一个新的 `shared_ptr` 指向这块内存，计数器加一；每当有一个 `shared_ptr` 消亡或被重新指向别处，计数器减一。当计数器归零（没人再需要这块内存了），自动 delete。

```cpp
shared_ptr<int> sp1 = make_shared<int>(42);   // 计数器 = 1
{
    shared_ptr<int> sp2 = sp1;                 // 计数器 = 2（拷贝引用计数增加）
    shared_ptr<int> sp3 = sp1;                 // 计数器 = 3
}   // sp2 和 sp3 出作用域，计数器回到 1
// sp1 仍然持有资源，安全

cout << *sp1 << endl;  // 42，依旧有效
```

`shared_ptr` 可以拷贝，拷贝多少次都行，引用计数会跟着走。只有当所有指向同一块内存的 `shared_ptr` 都消失了，那块内存才会被释放。

------

**make_shared：比你直接 new 更好**

```cpp
shared_ptr<int> sp1(new int(42));        // 可以，但不是最优
auto sp2 = make_shared<int>(42);         // 更推荐——一次分配完成，效率更高
```

`make_shared` 会在内部只做一次内存分配（同时保存对象本身和控制块），比先 `new` 再交给 `shared_ptr`（这是一个两次分配）更快。规则很简单：能用 `make_shared`/`make_unique` 就不要自己手写 `new`。

------

**shared_ptr 的循环引用问题**

看一个特殊的场景：

```cpp
class B;   // 向前声明

class A
{
public:
    shared_ptr<B> ptr;
};

class B
{
public:
    shared_ptr<A> ptr;
};

int main()
{
    auto a = make_shared<A>();
    auto b = make_shared<B>();

    a->ptr = b;    // a 指向 b，b 引用计数 +1
    b->ptr = a;    // b 指向 a，a 引用计数 +1

    // 离开 main 时：a 和 b 本地 shared_ptr 销毁，计数各减 1
    // 但它们相互持有对方的 shared_ptr，计数器永远不会到 0
    // => 内存泄漏！

    return 0;
}
```

这种情况叫**循环引用**。光靠 `shared_ptr` 无法解决——它会让两个对象"相依为命"，天荒地老都不放手。

解决方案：把其中一边换成 **`weak_ptr`**。

------

**weak_ptr：旁观的智能指针**

`weak_ptr` 是一个特殊的智能指针——它指向一块由 `shared_ptr` 管理的内存，但**不增加引用计数**。它就像一个"旁观者"——在一旁看着，但不对内存拥有所有权。

```cpp
class A
{
public:
    shared_ptr<B> ptr;        // 持有 B 的共享所有权
};

class B
{
public:
    weak_ptr<A> ptr;          // 旁观 A，不增加计数——打破循环引用
};
```

使用 `weak_ptr` 时，不能直接解引用。你得先"提升"为一个临时的 `shared_ptr`，确认指向的对象还活着：

```cpp
if (auto sp = weak.lock())    // lock() 返回一个临时的 shared_ptr：如果对象还在，就成功
{
    // 通过 sp 安全使用
}
else
{
    // 对象已经死掉了
}
```

`weak_ptr` 的其他用途还包括观察者模式、缓存淘汰机制等——凡是你想引用一个东西但不想妨碍它被销毁的场合，都可以考虑 `weak_ptr`。

------

**什么时候用什么**

```cpp
// 默认：unique_ptr
//   一块内存只有一个人管 → unique_ptr
auto obj = make_unique<MyClass>();

// 需要多个人共享时：shared_ptr
auto sp = make_shared<int>(42);
anotherFunction(sp);    // 传一份 shared_ptr，引用计数 +1

// 旁观：weak_ptr
//   解决循环引用 + 缓存/观察场景
```

对于 C++ 新手，一个极好的编程习惯是：**绝不在新的代码中写 raw new 和 raw delete**。优先使用 `make_unique` 和 `make_shared`，基本能覆盖 95% 的动态内存场景。`new`/`delete` 留给库作者和极高性能要求的底层代码去处理。
