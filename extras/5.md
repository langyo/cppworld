# 地心探索 · 内存分配控制

默认的 `new` 和 `delete` 给你拿来就用的堆内存，但对于某些真实接近硬件的场景——游戏引擎、实时系统、高频数据——开发者需要更精确地决定内存从哪来、怎么分配、何时释放。

------

## 重载类的 operator new / delete

你完全可以为某个特定类定制内存分配：

```cpp
#include <iostream>
#include <new>

class MyObject
{
    int data[100];     // 一个大对象，分配频繁会重伤通用分配器
public:
    static void* operator new(size_t size)
    {
        cout << "MyObject::new，要求 " << size << " 字节\n";
        return ::operator new(size);   // 归底层全局 new 分配原始内存
    }

    static void operator delete(void* ptr)
    {
        cout << "MyObject::delete\n";
        ::operator delete(ptr);
    }
};

int main()
{
    MyObject* obj = new MyObject();     // 输出 "MyObject::new，要求 400 字节"
    delete obj;                         // 输出 "MyObject::delete"
}
```

这只是一个最简的日志包装，实际用途完全不止于打印——你可以把 `operator new` 指向自己造的内存池、对齐存储、或加性能统计。

------

## 分配器：让容器也用上你的定制内存

标准库容器的内存管理不是写死的——`vector`、`map` 等都可以通过**自定义分配器**指定从哪块内存源出货：

```cpp
#include <iostream>
#include <vector>
#include <memory>

template<typename T>
class LoggingAllocator : public allocator<T>
{
public:
    T* allocate(size_t n)
    {
        cout << "分配 " << n << " 个元素\n";
        return allocator<T>::allocate(n);
    }

    void deallocate(T* p, size_t n)
    {
        cout << "释放 " << n << " 个元素\n";
        allocator<T>::deallocate(p, n);
    }
};

int main()
{
    vector<int, LoggingAllocator<int>> v;
    v.push_back(42);       // 会在后台打印分配日志
}
```

分配器是高级话题——完整的自定义分配器还需要处理 `rebind`、`propagate_on_container_move_assignment` 等一大堆细节。这里只是要你知道标准库的容器内存来源是你可控的，在游戏引擎、嵌入式系统、高性能数据库里，分配器是完全替换内存后台的主入口。

------

## 内存池：自己当批发商

内存池的核心思想是**你自己先一口气向操作系统申请一大片内存，然后应用层再找你零零碎碎地"批发"**，而不用每次调 `new` 都去跟操作系统讲价。

下面是一个简单但完整可用的**单调递增（bump）分配器**——只分不发，直到整个帧/周期结束一块儿还原：

```cpp
#include <iostream>
#include <new>

class BumpAllocator
{
private:
    char*  buffer;
    size_t capacity;
    size_t offset = 0;

public:
    explicit BumpAllocator(size_t size)
        : buffer(new char[size]), capacity(size) {}

    ~BumpAllocator() { delete[] buffer; }

    void* allocate(size_t size, size_t alignment = alignof(max_align_t))
    {
        // 对齐计算：保证返回地址是 alignment 的倍数
        uintptr_t addr = reinterpret_cast<uintptr_t>(buffer + offset);
        uintptr_t aligned = (addr + alignment - 1) & ~(alignment - 1);
        size_t padding = aligned - addr;

        if (offset + padding + size > capacity)
            throw bad_alloc();

        offset += padding + size;
        return reinterpret_cast<void*>(aligned);
    }

    void reset() { offset = 0; }    // 一键清空，不用逐个 delete
};

int main()
{
    BumpAllocator pool(1024);       // 预分配 1024 字节的大块

    int* a = static_cast<int*>(pool.allocate(sizeof(int)));
    int* b = static_cast<int*>(pool.allocate(sizeof(int)));
    double* c = static_cast<double*>(pool.allocate(sizeof(double)));

    *a = 10;
    *b = 20;
    *c = 3.14159;

    cout << *a << " " << *b << " " << *c << "\n";

    pool.reset();     // 全部还回去了
}
```

这种分配器在游戏的每个渲染帧里大量使用——一帧开始时 offset 归零，帧内所有临时对象从上面疯狂申请，帧结束后一把 reset，全程没有一次独立的 `delete` 开销。

------

## placement new：在指定地址上构造对象

`new` 默认既分配内存又构造对象。有时候内存已经在那里了——比如上面分配器的 `allocate` 已经给了地址——你只缺"在那地址上跑一次构造函数"。

这就是 `placement new` 的用场：

```cpp
#include <new>

int main()
{
    char buffer[sizeof(int)];                  // 提前准备好内存
    int* p = new (buffer) int(42);             // 在 buffer 位置构造 int 值 42
    cout << *p << "\n";
    p->~int();                                 // placement new 的对象你必须手动析构
}
```

`new (address) Type(args)` 不去分配新内存，只在已有地址上执行 Type 的构造函数。析构也不能依赖自动机制——你得显式调 `p->~Type()`。

------

这些对大多数应用程序员来说只是冷兵器——大多数时候你不需要手写分配器、内存池或 placement new。但知道 C++ 把"内存从何而来、何时释放、怎样对齐"这全部链条都留了更换接口给你，才真正理解"性能"这门功夫 C++ 不靠隐藏魔术，而是把机制一层层摊开让你接手的哲学。
