# 启动《Minecraft》 · 系统调用库

有些操作不是 C++ 语言或者标准库能直接做到的——比如列出目录下的所有文件、启动另一个程序、读系统环境变量、网络请求。这些需要借助操作系统的"系统调用"——直接跟操作系统签账务窗的道。

C++ 本身没有完整封装这些，但支持标准化的部分在 `<filesystem>`（C++17）、`<cstdlib>` 和平台原生的 API（如 Windows API 和 POSIX）中。

------

**`std::filesystem`（C++17）**

遍历目录、创建文件夹、删除文件、检查文件是否存在——全都用一致的可移植代码：

```cpp
#include <filesystem>
#include <iostream>
namespace fs = std::filesystem;

int main()
{
    fs::path p = fs::current_path();     // 获取当前工作目录
    cout << "当前目录: " << p << endl;

    // 遍历当前目录
    for (const auto& entry : fs::directory_iterator(p))
        cout << (entry.is_directory() ? "[DIR]  " : "[FILE] ") << entry.path().filename() << endl;

    // 创建目录
    fs::create_directory("output");

    return 0;
}
```

`fs::path` 是文件名和路径的多平台表示，它自动处理 Windows 上的 `\` 与 Linux/macOS 上的 `/` 之间的差异。`directory_iterator` 提供了一种像遍历容器一样的体验——"一个目录中的所有条目"就是一个你可以用范围 for 逐一遍历的东西。 

此外，`fs::file_size`、`fs::remove`、`fs::copy` 等函数都已经标准化——写跨平台的文件工具不用再小心翼翼去 `#ifdef _WIN32` 了。

------

**启动外部程序（`std::system` 与 `popen`）**

用 `std::system` 可以运行一个外部命令行命令：

```cpp
#include <cstdlib>
std::system("echo Hello from system!");
```

`std::system` 在不同的操作系统上会走不同的底层实现（Windows 上是 `cmd`，POSIX 上是 `sh`），而且它的返回值不一定能清楚表达被调进程的退出码——这是它比较粗犷的原因。

如果需要更精细的控制——比如获取该程序输出结果、持续读标准输出——用 POSIX 的 `popen` / `pclose`（Linux/macOS）或 Windows 的 `_popen`，或者用 `boost::process` 库（C++ 标准库目前仍没有线程安全的子进程管理接口）。

------

**环境变量**

```cpp
#include <cstdlib>
const char* value = std::getenv("PATH");
if (value) cout << "PATH 的值为: " << value << endl;
```

`getenv` 返回平台环境变量的指针——不能修改，不需要释放。C++17 暂未提供 `setenv` 式变量的标准写法。

------

**网络**

标准 C++ 到目前为止没有内置 HTTP 或 Socket 的网络库。你需要借助操作系统特定接口（如 Linux 的 `socket`/`connect`、Windows 的 Winsock），或第三方跨平台库（如 Boost.Asio、cpp-httplib、libcurl）。

C++23 终于引入了 `std::net`，将基础的网络编程加入标准库，使其与 `<filesystem>` 一样成为可移植的系统接口——但在实际项目大量可用之前仍需要一些时间来成熟和普及。
