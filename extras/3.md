# 同时进行的任务 · 并行与并发库

摩尔定律的免费午餐吃完了——单核主频不再快速上涨，但核心数量翻来覆去地涨。如果你想让程序跑得快，**你必须学会利用多核**——这就是并发编程的价值所在。

C++11 将多线程完全纳入标准库（`<thread>`、`<mutex>`、`<condition_variable>`、`<future>`、`<atomic>`），让跨平台并发变成可移植的代码，而不是各种平台的私货线程库的乱炖。

------

**创建一个线程**

```cpp
#include <iostream>
#include <thread>
using namespace std;

void hello() { cout << "来自线程的问候" << endl; }

int main()
{
    thread t(hello);       // 创建线程，立即开始执行 hello
    t.join();              // 等待线程执行完毕
    return 0;
}
```

创建 `thread` 对象时传入一个可调用对象（函数、函数对象、Lambda），线程立即启动。`join` 在主线器等待子线程收工——如果不写 join 也不 detach，thread 的析构会调 `terminate`，整个程序因此崩溃。

`detach` 也有它的用途——某些长期驻留的后台线程可以分离后自己跑自己的，但这类线程必须保证它用的资源在使用期间一直有效（否则主线程退出后全局对象析构，detach 的线程还在继续读那些已死对象，崩溃难以调试）。

------

**互斥量：保护共享数据锁**

多个线程同时读写同一份数据会引发**数据竞争**——这是并发编程中最让人头疼也是最常见的 bug 类型。

```cpp
#include <mutex>

int counter = 0;
mutex mtx;

void increment()
{
    lock_guard<mutex> lock(mtx);   // 进入作用域自动加锁，离开作用域自动解锁
    ++counter;
}

int main()
{
    thread t1(increment);
    thread t2(increment);
    t1.join();
    t2.join();
    cout << counter << endl;   // 2，安全无竞争
}
```

`lock_guard`（C++11）和 `scoped_lock`（C++17 起推荐，支持多锁同时上锁避免死锁）遵守 RAII 原则——加锁解锁自动化，绝不忘记。任何时候写并发代码，遇到 `mutex` 就应该同时想到 `scoped_lock`，几乎不要手动调用 `mtx.lock()` / `mtx.unlock()`。

`std::shared_mutex` 和 `std::shared_lock`（C++17）提供读写锁——多个线程同时可以读、写操作独占。更多并发场景可以查阅 `<atomic>` 的无锁原子操作实现高性能计数器或自旋锁。

------

**原子操作（atomic）**

`atomic<int>` 提供"被原子保证的整数操作"——在硬件层面保证读-修改-写不会被其它线程插队。

```cpp
#include <atomic>

atomic<int> counter = 0;
// 免锁自增，比 mutex 版本快得多
counter.fetch_add(1);
```

原子操作是无锁编程的基石——无锁队列、无锁链接表等高性能数据结构都建立在它之上。但它不是银弹：逻辑越复杂，用原子操写就越容易出错，绝大多数场景用 mutex + scoped_lock 即可。

------

**std::async 与 future**

如果你只是想在一个独立线程里执行某个任务，然后拿到结果，`async` 把"创建线程、取回结果"这件事做成了一个看起来像同步调用的包装：

```cpp
#include <future>

int compute() { return 42; }

int main()
{
    future<int> result = async(launch::async, compute);  // 异步执行
    cout << result.get() << endl;                         // 等待并拿到结果

    return 0;
}
```

`launch::async` 强制另起线程；`launch::deferred` 则延迟到调用 `get()` 时再在当前线程执行（惰性求值）。如果你不确定哪种策略合适，可以干脆省略 launch 参数，让系统自己选择。

`async` + `future` 对于"丢一个任务到另一个线程，等结果"这种直线并行的需求来说，比手动创建线程干净省事。
