# 与硬件背靠背 · C++、C语言与汇编的协同

C++ 的一大能力是能方便地调用 C 语言的库，甚至能在 C 语言中调用 C++ 的对象（用一些巧妙手法）。有时候，为了性能，你也可以在 C++ 里嵌一小段汇编。

------

**从 C++ 调用 C**

C 的函数没有名字修饰（name mangling），而 C++ 默认会对函数名进行修饰以支持重载。为了让 C++ 代码能调用一个 C 编译产出的函数，你需要用 `extern "C"` 告诉编译器："这批函数按 C 的链接习惯来"。

```cpp
extern "C"
{
    #include "some_c_library.h"
}

// 或作用于单个函数
extern "C" int legacyFunction(int x);
```

如此，C++ 编译器就不会对 `legacyFunction` 加上类型修饰，保证链接时能找到 C 目标文件里那个没有修饰的符号。

------

**从 C 调用 C++**

在 C++ 这边用 `extern "C"` 包装暴露给外界的函数：

```cpp
// CppLib.cpp
extern "C"
{
    int compute(int a, int b) { return a * b + a; }
}
```

C 那边只需声明：

```c
// main.c
extern int compute(int a, int b);
```

链接时同时把 C 和 C++ 的目标文件放在一起即可。但 C 无法直接使用 C++ 类的对象——这是语言语义层级的差距，中转必须经过 `extern "C"` 包裹的全局函数。典型的封装方式是：C++ 侧定义一个 C 风格的 Handle（不透明的 `void*`），对外的 C API 里传递这个 Handle，内部再 `reinterpret_cast` 回到 C++ 对象成员函数。

------

**内联汇编**

C++ 编译器支持 `__asm__` 或 `asm` 关键字（因编译器而异），允许你在 C++ 中间插入汇编指令。这是给编译器优化兜底的最后手段——通常是因为你需要用硬件指令（如 CPUID、SIMD 特定指令）来做标准语法不支持的事。

```cpp
#ifdef _MSC_VER
__asm { mov eax, 1 }
#else
__asm__("movl $1, %eax");
#endif
```

内联汇编的行为因架构、编译器而异——你在 x86 的 MSVC 上写的东西，拿到 ARM 的 Clang 上连编译都过不去。所以只有在"没有可移值的 C++ 替代实现"时，才拿出汇编。

------

**调用约定**

当 C++ 和 C 互相调用时，调用约定必须一致。`extern "C"` 默认采用平台默认的 C 调用约定（通常是 `cdecl` 或类似方式），不需要额外标注。但如果你手动声明函数指针或调用 DLL，用 `__cdecl`、`__stdcall` 等调约说明符显式匹配：

```cpp
extern "C" __stdcall int WinApiFunction(int x);
```

调用约定不匹配的最典型症状是栈被破坏后函数返回时崩溃——在 Windows 跨模块调用动态库时尤其需要反复核对。

------

**C++ 的类型和 C 类型的对应**

C++ 的标准库类型（如 string、vector）在 C 中不可用。想要跨边界传递字符串，要传递 `const char*`；想传递数组，传 `T*` 再加一个显式的长度参数。这是互操作中的障碍，也是很多 C 库的封装层都要写一大坨转换代码的根源。
