# 牛奶还是奶牛 · 位域与字节序

计算机存储数字还有两个你不得不面对的底层问题：**位域**让你在一个字节内打包多个小数据片段；**字节序**决定了多字节数据在内存中是"高尾在前"还是"低尾在前"。

------

**位域：一个字节当多个用**

```cpp
struct Flags
{
    uint8_t read    : 1;   // 占 1 位
    uint8_t write   : 1;
    uint8_t execute : 1;
    uint8_t         : 5;   // 填充——占 5 位，没名字（匿名位域）
};
// sizeof(Flags) 通常是 1 字节
```

位域让你把多个小信息挤进同一个整数类型的内存空间里。它在读取硬件寄存器、解析二进制协议头、或节省大规模标志数组的内存占用时非常有用。

注意：位域的布局（从左往右还是从右往左、跨越字节边界的规则等）由编译器决定，两个不同的编译器可能给出不同的排列，所以用它作二进制序列化时一定要谨慎。

------

**位域的局限性**

- 不能对位域取地址（因为里面的位没有自己独立的地址）。
- 不能有引用指向位域（绑定不到位这个级别）。
- `sizeof` 不能直接应用于单独的位域成员。
- 其跨平台的二进制布局不可移植。

如果你需要一整个字节的"位级键组"，`std::bitset` 往往比位域结构体更安全：它封装好了位操作，抽象度高一级，也更可读。

------

**字节序：大端 vs 小端**

一个 4 字节的 `int` 在内存中的排列有两种主流方式：

```
整数 0x12345678

大端 (Big Endian)  —— 高位在前
+----+----+----+----+
| 12 | 34 | 56 | 78 |
+----+----+----+----+

小端 (Little Endian) —— 低位在前
+----+----+----+----+
| 78 | 56 | 34 | 12 |
+----+----+----+----+
```

x86 架构为小端，绝大多数现代 PC 都是小端。网络协议（TCP/IP）使用大端——即"网络字节序"。很多文件格式（如 JPEG）会用大端存储多字节值以保证跨平台可读性。

------

**检测字节序**

```cpp
bool isLittleEndian()
{
    uint16_t x = 1;                          // 在内存里要么是 01 00 (小端) 要么 00 01 (大端)
    return *reinterpret_cast<uint8_t*>(&x) == 1;
}
```

这个技巧利用了"相同数据在内存中的不同起步位"：小端的数字 1 在第一次字节里就是 1；大端的数字 1，第一次字节是 0，1 在后面的位置。

------

**何时需要处理字节序**

- 接收网络上发来的数据（`ntohl`/`htonl` 等函数做转换）。
- 读写某些特定格式的二进制文件（如 BMP 采用小端、PNG 头采用大端）。
- 跨平台数据交换——你自己定义二进制协议时先确定是"协议端序"还是"原汁原味"存放。

现代很多序列化库（Protobuf、Cap'n Proto、FlatBuffers 等）已经帮你解决了字节序和布局的麻烦，你一般不需要手排字节——但如果没有任何库、需要裸写二进制协议，就别忘了这一步。而知道这些东西的存在，也是在出现跨平台二进制 bug 的时候能快速定位的底气。
