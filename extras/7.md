# 牛奶还是奶牛 · 位域与字节序

数据在内存中的物理表示还有两个你迟早要面对的细节：**位域**让你把一个字节掰碎了用，几个 bit 就能代表一个小信息；**字节序**决定了多字节数据类型在内存中是从高位开始存还是从低位开始存。

------

## 位域：一个字节当多个字段用

有些信息根本不需要一个完整的 `int` 来存——比如一个标志位只需要 0 或 1。位域让你在一段已知大小的整数类型空间内，把不同的小信息打包在一起：

```cpp
#include <cstdint>
#include <iostream>

struct Permissions
{
    std::uint8_t read    : 1;    // 占 1 个二进制位
    std::uint8_t write   : 1;
    std::uint8_t execute : 1;
    std::uint8_t         : 5;    // 匿名位域——填充剩余 5 位
};

int main()
{
    std::cout << "sizeof(Permissions) = " << sizeof(Permissions) << "\n";  // 通常为 1

    Permissions p{};
    p.read    = 1;
    p.write   = 1;
    p.execute = 0;

    if (p.read && p.write)
        std::cout << "可读可写\n";    // 位域当成 bool 用，0/1 不可溢出

    return 0;
}
```

这个结构体才占 **一个字节**。如果没有位域，你很可能要用三个 `bool` 或三个 `int`，花了 3 字节以上，还得自己做位运算去组合拆解。

位域在使用上有几条硬性限制：
- 不能对位域取地址（它没有独立的字节地址）。
- 不能用引用绑定到位域。
- 位域的跨字节边界和对齐由编译器决定——**不同编译器可能排出不同的结果**，不能依赖它做跨平台二进制序列化。

如果你需要打包更多的位或更复杂的位操控，`std::bitset<N>`（N 在编译期确定）通常比手写位域更安全可读。

------

## 字节序：大端 vs 小端

一个四字节的 `int` 在内存里是怎么排列的？两种主流方式：

```
整数 0x12345678

大端（Big Endian）——高位字节放在低地址
  地址  +0    +1    +2    +3
       [0x12][0x34][0x56][0x78]

小端（Little Endian）——低位字节放在低地址
  地址  +0    +1    +2    +3
       [0x78][0x56][0x34][0x12]
```

x86 处理器是小端，大多数现代 PC 和一部分 ARM 配置也是小端。但**网络传输**用的是大端（"网络字节序"），很多二进制文件格式（如 JPEG）也用大端。

------

## 检测你的机器是大端还是小端

```cpp
#include <cstdint>
#include <iostream>

bool isLittleEndian()
{
    std::uint16_t x = 1;    // 占用 2 个字节，数值为 0x0001
    // 小端：(低地址) 01 00    大端：(低地址) 00 01
    return *reinterpret_cast<unsigned char*>(&x) == 1;
}

int main()
{
    if (isLittleEndian())
        std::cout << "本机是小端\n";
    else
        std::cout << "本机是大端\n";
}
```

`reinterpret_cast<unsigned char*>(&x)` 强制以字节视角去看 `x` 的第一个字节。小端时它是 `01`（等于 1），大端时它是 `00`（等于 0）。

C++20 提供了标准化的 `std::endian` 枚举，可以编译期直接用 `if constexpr (std::endian::native == std::endian::little)` 来判断——不需要手写上面的魔法代码。

------

## 什么时候要关心字节序

- 接收网络上发来的数据，需用 `ntohl`、`htonl`（POSIX）函数将其从网络字节序转为主机字节序。
- 读写二进制文件——有些格式写死了某一种端序。
- 自己设计跨平台二进制协议时，先确定"协议端序"是固定大端还是随机器，然后写序列化/反序列化逻辑。

现代大多数时候你不需要手动处理字节序——Protobuf、Cap'n Proto、FlatBuffers 等序列化框架已经替你把端序和布局都消化干净了。但如果你在写原始二进制协议库、或者调试跨平台收发数据时发现"数字读回来怎么不对"，知道大端小端是什么，就是你从迷宫走出来的那张一直握在手里、但此时才看得见的地图。
