# 分化方式 · RTTI（运行时类型信息）

编译器在编译你的代码时，对每个变量的类型都一清二楚。但在运行期，当你手里只有一根基类指针，你想知道它到底指向的是哪个子类对象，怎么办？

答：依靠 **RTTI**（Run-Time Type Information，运行时类型信息）。C++ 提供两个核心工具：`typeid` 和 `dynamic_cast`。

------

## typeid：查到你的真名

先用 `typeid` 来探一探运行时的真实身份：

```cpp
#include <iostream>
#include <typeinfo>

class Animal
{
public:
    virtual ~Animal() = default;    // 有虚函数的类才是多态类——没有这行，RTTI 就不工作
};

class Cat : public Animal {};
class Dog : public Animal {};

int main()
{
    Cat cat;
    Dog dog;

    Animal& a1 = cat;
    Animal& a2 = dog;

    // typeid 在多态上下文中返回的是实际对象的类型
    cout << "a1 的真实类型: " << typeid(a1).name() << "\n";
    cout << "a2 的真实类型: " << typeid(a2).name() << "\n";

    // 直接比较
    if (typeid(a1) == typeid(Cat))
        cout << "a1 确实是一只猫\n";

    if (typeid(a2) != typeid(Cat))
        cout << "a2 不是猫\n";

    return 0;
}
```

`typeid` 返回一个 `std::type_info` 对象（定义在 `<typeinfo>` 中），可以直接用 `==` 和 `!=` 比较类型是否相同。

一个常见的坑：如果你写 `typeid(a1)` 时 `a1` 不是引用而是指针，拿到的是指针本身的类型（如 `Animal*`），而非指向的对象的类型。所以**记得解引用**或者**用引用**。

还有一点：如果类连一个虚函数都没有，`typeid` 就没有运行时信息可查了——它只会返回静态声明的类型。只有多态的类（带 `virtual` 的）才能享受正宗的 RTTI。

------

## dynamic_cast：安全的向下转型

你有基类指针，想看看它是不是某种特定的派生类。`dynamic_cast` 在运行时执行类型检查，如果匹配就返回目标类型指针，不匹配就返回 `nullptr`：

```cpp
#include <iostream>
#include <vector>
#include <memory>

class Animal
{
public:
    virtual ~Animal() = default;
    virtual void speak() const = 0;    // 纯虚函数——动物都会叫
};

class Cat : public Animal
{
public:
    void speak() const override { cout << "喵~\n"; }
    void scratch() const { cout << "猫抓了你一下！\n"; }
};

class Dog : public Animal
{
public:
    void speak() const override { cout << "汪！\n"; }
    void fetch() const { cout << "狗去捡球了！\n"; }
};

int main()
{
    vector<unique_ptr<Animal>> pets;
    pets.push_back(make_unique<Cat>());
    pets.push_back(make_unique<Dog>());
    pets.push_back(make_unique<Cat>());

    for (const auto& pet : pets)
    {
        pet->speak();     // 多态——猫喵狗汪，自动分派

        // 尝试向下转型为 Cat
        Cat* c = dynamic_cast<Cat*>(pet.get());
        if (c)
            c->scratch();  // 只有猫才执行这里

        // 尝试向下转型为 Dog
        Dog* d = dynamic_cast<Dog*>(pet.get());
        if (d)
            d->fetch();    // 只有狗才执行这里
    }

    return 0;
}
```

输出：

```
喵~
猫抓了你一下！
汪！
狗去捡球了！
喵~
猫抓了你一下！
```

`dynamic_cast<T*>(ptr)` 在 `ptr` 确实指向一个 `T` 类型的对象时返回有效指针，否则返回 `nullptr`。它背后沿着继承树查表，开销比普通指针转换大一些，但能帮你兜住"类型不对"的错误。

如果用引用版 `dynamic_cast<T&>(ref)`，碰到类型不匹配时会抛出 `std::bad_cast` 异常——这在"理论上一定能转对、但以防万一把它把关住"的场景里反倒更合适。

------

## 什么时候不该用 RTTI

RTTI 虽好，但你的代码里如果频繁出现类似这样的东西：

```cpp
if (typeid(*p) == typeid(Cat))
    static_cast<Cat*>(p)->scratch();
else if (typeid(*p) == typeid(Dog))
    static_cast<Dog*>(p)->fetch();
else if (typeid(*p) == typeid(Bird))
    // ...
```

那你八成是在和多态作对——虚函数的本意就是让你不要手写这种"判断类型再分派"的链式 if。一个好的面向对象设计会让每种派生类把自己的独特行为封装在虚函数的覆盖版本里，而不是让你在调用方枚举类型再选择路径。

RTTI 该在什么场景浮现？当你写的代码需要"跨类型边界"的时候——例如序列化/反序列化框架、插件系统中动态加载的 DLL 里的类实例识别、或者写一个通用的"对象诊断工具"时。日常的业务代码中，`dynamic_cast` 偶有登场（要检查能否转换为某个具体子类），`typeid` 则更少见。

------

## RTTI 的开销与禁用

RTTI 不是无代价的——编译器需要为每个多态类维护一张 `type_info` 表，`dynamic_cast` 需要沿继承链向上向下遍历。在嵌入式或一些性能极度敏感的系统中，可能会用编译选项（gcc 的 `-fno-rtti`，msvc 的 `/GR-`）完全关闭 RTTI。关闭之后 `typeid` 和 `dynamic_cast` 不能用了，但二进制体积会缩小，虚函数仍可正常工作。
