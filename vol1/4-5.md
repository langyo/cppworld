# 缓兵之计 · 异常处理语句

写程序嘛，意外总是在所难免。

用户可能在应该输数字的地方输了一串汉字；文件可能早就被隔壁老王删掉了；网络说断就断，跟心情好不好完全没关系。

遇到这些情况，程序要是直接崩了，那体验可就太差了。我们需要一种机制，能够体面地处理意外，要么让程序接着跑，要么至少给用户一句像样的人话，而不是糊用户一脸报错代码。

C++ 的**异常处理机制**就是为此而生的。

------

先来看一个很朴素的问题：写个除法函数。

```cpp
int divide(int a, int b)
{
    return a / b;
}
```

这个函数看起来没什么问题——只要 `b` 不为 0。可是如果调用的人手一滑传了个 0 进来呢？在数学中，除以零是没有定义的，程序走到这里就出错了。

传统的做法是在函数里加检查：

```cpp
int divide(int a, int b)
{
    if (b == 0)
    {
        cout << "错误：除数不能为零！" << endl;
        return 0;  // 返回一个"看起来没事"的默认值
    }
    return a / b;
}
```

但这样做有个很要命的缺陷：调用这个函数的人根本分不清返回的 0 是"10 除以 2 的真实结果"还是"出错了随便塞给你的"。他得额外写一堆判断才能搞清楚状况，代码又臭又长。

------

C++ 的异常处理机制提供了一个更干净的解决思路。它靠三个关键字来干活：`try`、`catch` 和 `throw`。

- **throw**：当遇到异常情况时，朝上空丢一个"信号弹"。
- **try**：把可能出问题的代码圈起来，对编译器说："这片区域帮我盯着点，可能有异常。"
- **catch**：守在下面接住"信号弹"，决定怎么处理。

我们用这套机制重写刚才的例子：

```cpp
int divide(int a, int b)
{
    if (b == 0)
        throw "除数不能为零！";     // 抛出异常，函数到这里直接截断
    return a / b;
}

int main()
{
    try
    {
        int result = divide(10, 0);  // 这里会抛出异常
        cout << "结果: " << result << endl;
    }
    catch (const char* msg)           // 接住异常，msg 就是抛上来的内容
    {
        cout << "捕获到异常: " << msg << endl;
    }

    cout << "程序继续执行……" << endl;
    return 0;
}
```

打印出来的结果是：

```
捕获到异常: 除数不能为零！
程序继续执行……
```

程序没有炸，而是稳稳当当地处理了异常，然后接着往下跑。

------

我们分别细看一下这三个关键字。

**throw**

`throw` 用于抛出异常。后面可以跟任意类型的值：

```cpp
throw 42;               // 抛个整数
throw "出错了！";        // 抛个 C 风格字符串
throw 3.14;             // 抛个浮点数
throw 自定义异常();      // 抛个自定义的异常对象（后面会讲）
```

`throw` 一旦执行，函数当场截断，程序开始向上寻找能接住这个异常的 `catch` 块。这一点非常关键：`throw` 后面的代码**不会被执行**。

**try**

`try` 块包住那些可能炸的代码：

```cpp
try
{
    // 这里放可能产生异常的代码
    危险的操作();
}
```

如果 `try` 块里的代码抛出了异常，程序会立刻跳出 `try` 块，去找下面匹配的 `catch` 块。`try` 块中还没执行的代码就再也不会被执行了。

**catch**

`catch` 块接住异常并处理：

```cpp
catch (异常的类型 变量名)
{
    // 处理异常的代码
}
```

`catch` 块必须紧跟在 `try` 块后面。一个 `try` 后面可以挂多个 `catch`，分别接不同类型的异常：

```cpp
try
{
    // 可能产生异常的代码
}
catch (int e)
{
    cout << "接到了整数异常: " << e << endl;
}
catch (const char* e)
{
    cout << "接到了字符串异常: " << e << endl;
}
catch (...)        // 三个点 = 什么类型的异常都接
{
    cout << "接到了某种我不知道类型的异常" << endl;
}
```

`catch(...)` 是万能接盘侠，什么类型都能接，一般放在最后兜底。

------

实战中我们一般用标准库提供的异常类，而不是随手抛个字符串。标准库在 `<stdexcept>` 里定义了一套现成的异常：

| 异常类 | 什么时候用 |
| ------ | ---------- |
| `exception` | 所有标准异常的基类 |
| `runtime_error` | 运行时才能发现的错误 |
| `logic_error` | 能提前预判的逻辑错误 |
| `invalid_argument` | 参数无效 |
| `out_of_range` | 超出范围 |
| `overflow_error` | 算术溢出 |

这些异常类都有一个 `what()` 成员函数，能返回一段描述异常的字符串。用法如下：

```cpp
#include <iostream>
#include <stdexcept>
using namespace std;

int divide(int a, int b)
{
    if (b == 0)
        throw invalid_argument("除数不能为零！");
    return a / b;
}

int main()
{
    try
    {
        int result = divide(10, 0);
        cout << "结果: " << result << endl;
    }
    catch (const invalid_argument& e)
    {
        cout << "无效的参数: " << e.what() << endl;
    }
    catch (const exception& e)
    {
        cout << "出了某种异常: " << e.what() << endl;
    }
    return 0;
}
```

注意，捕获的时候我们写的是 `const invalid_argument&`，用引用来接，而不是直接拷贝——这样效率更高，而且能保留异常对象的完整信息。

------

异常处理虽然好用，但它是有额外开销的，不能滥用到哪哪都 `try-catch`。记住几条原则：

- 异常应用来处理**真正的意外情况**，不要用它替代正常的 `if-else` 判断。
- 如果一个错误是可以预见的、容易就地恢复的，用返回值判断就好，比抛异常快得多。
- 抛出异常时应该带上足够的信息，让接的人知道发生了什么。
- 接异常时要尽可能接具体的类型，别动不动就用 `catch(...)` 一把梭——你总得知道到底出了什么问题才好对症下药。

------

来看一个更实际的例子——让用户输入一个整数：

```cpp
#include <iostream>
#include <stdexcept>
using namespace std;

int getInteger(const string& prompt)
{
    cout << prompt;
    int value;
    cin >> value;

    if (cin.fail())               // 用户输入的根本不是数字
    {
        cin.clear();              // 清除错误标记
        cin.ignore(1000, '\n');   // 丢弃掉垃圾输入
        throw invalid_argument("请输入一个有效的整数！");
    }

    return value;
}

int main()
{
    while (true)
    {
        try
        {
            int num = getInteger("请输入一个整数: ");
            cout << "你输入的是: " << num << endl;
            break;                   // 输入正常，结束循环
        }
        catch (const invalid_argument& e)
        {
            cout << "错了: " << e.what() << " 再试一次。" << endl;
        }
    }

    return 0;
}
```

用户只要不老实输入整数，程序就会一直请他重新输入，直到老实为止。
